# Barebones 2025 stateful spec compatibility

Backlog entry. Captures our reflexions on retrofitting support for the current
2025 stateful MCP spec (2025-06-18 / 2025-11-25) on top of the stateless
2026-07-28 core. We do not know yet if we will do it.

## Purpose and scope

The only goal is to **keep existing 2025 tool calls working while we migrate
our servers** to the stateless core. This is a migration shim, not feature
parity:

- Supported: `initialize` handshake, `notifications/initialized`, `tools/list`,
  `tools/call` (with the usual progress/log notifications on the POST's own SSE
  response), and whatever list/read/get methods our migrating servers actually
  use.
- Everything stateful-fancy is out: no resumability (`Last-Event-ID`), no
  server-initiated requests (sampling/elicitation), no `resources/subscribe`.
- Tool `{:input_required, ...}` returns (MRTR) have no 2025 equivalent. On the
  compat transport this is an **invalid callback return** — a server-side
  contract violation, not a client fault. The worker crashes (as for any
  invalid return), and the transport converts the crash into a JSON-RPC
  internal error, the same way the existing `:server_crashed` path does. The
  client must not be told its request was faulty.
- 2026-only methods (`server/discover`, `subscriptions/listen`) are rejected
  naturally: the 2025 validator does not know them.

The `main` branch still holds the full pre-fork stateful implementation and is
the reference / salvage source for all of this.

## Current choices

### Vocabulary: regenerated `V2511` entities

Reintroduce the old vocabulary as a generated `GenMCP.MCP.V2511` module set
(2025-11-25 schema, which also serves 2025-06-18 clients), living next to
`V2607`. The schema codegen tooling already supports versioned namespaces, so
this part is mechanical. A `V2511` validator selects the 2025 request surface.

### Second transport layer

A separate legacy plug (e.g. `GenMCP.Transport.StreamableHTTP.V2511`) rather
than branching inside the current one: header rules, the `initialize` special
case, and session lookup differ too much. The SSE machinery
(`stream_loop`/keepalive/finalize in the current `Impl`) is version-agnostic
and gets extracted and shared. Requests are translated at this boundary into
the per-request worker contract the stateless core already runs, so the Suite
and the provider behaviours stay single-sourced.

If both protocols must share one URL, a thin dispatcher plug sniffs
(`MCP-Protocol-Version: 2026-07-28` header vs `initialize` body) and delegates.

### Simplified session controller

A small behaviour with two operations: **create** on `initialize`, **fetch** on
every subsequent request. Session state is written once and only read
afterwards — it stores the client initialization parameters the server actually
reads later, nothing more.

### Data-bearing session id (default controller)

The default implementation stores nothing server-side: the `Mcp-Session-Id`
**is** the session data, sealed with `GenMCP.Token` (already in the codebase,
cluster-safe via `secret_key_base`). Fetch = decrypt. `DELETE` is a no-op
accept. No revocation; sessions expire by token TTL only.

To keep the header small (budget: cookie-sized, a few hundred bytes — well
under proxy per-header limits), we encode a pruned tuple, not the full
initialize params (`clientInfo` is unbounded: descriptions, icon data URIs...):

    {1, negotiated_vsn, client_name, client_version}

The leading `1` is a mere 8-bit integer tag. Later gen_mcp releases will not
have session ids at all, so no versioning scheme is needed beyond this.

Deployments that need the full `clientInfo`/capabilities server-side, or real
revocation, plug their own storage-backed controller via the behaviour. Since
the default holds no process and no table, the earlier idea of an optional
`:syn` dependency for an in-memory default is superseded — nothing needs a
registry.

### GET stream

Plan A is to check whether our migrating clients hold the GET stream open at
all: the 2025 spec allows answering `405 Method Not Allowed`, and tool-call
progress/logs ride the POST response stream, not the GET one. If some client
does need it, redirect GET to the subscription handler (or a similar ad-hoc
delivery hook).

## Release strategy

- The existing stateless code ships as **v1**.
- This retrofit, if done, ships as **v2** (stateless core + compat layer).
- In parallel, a **temporary `gen_mcp2` hex package** with all modules renamed
  `GenMCP2.*`. Rationale: Mix cannot install two versions of one package and
  both claim the `GenMCP.*` namespace, so without it a dep bump forces porting
  every endpoint in one commit. `gen_mcp2` allows endpoint-by-endpoint
  migration inside one app: implement new services against the `2` package,
  then once the old dependency is removed, mass-rename `GenMCP2` → `GenMCP`
  and swap the dep — it lands on identical code.

`gen_mcp2` is generated, never hand-maintained: a release script textually
rewrites `GenMCP` → `GenMCP2` and `:gen_mcp` → `:gen_mcp2` across the tree and
runs the full test suite on the renamed copy. Name-load-bearing spots to audit:
telemetry event prefixes, the registered `GenMCP.ServerSupervisor` name (both
apps must boot in one VM), `Application.get_env(:gen_mcp, ...)` keys. Publish
with a "temporary migration package" notice, version-locked to the `gen_mcp`
v2 release it is generated from; retire after our migrations.

## Open questions

- `logging/setLevel`: the token session is immutable, so it cannot be stored.
  Accept-and-ignore vs JSON-RPC error.
- Exact 2025 method surface to enable in the `V2511` validator (drive it from
  what the migrating servers use, not from the spec).
- Default token TTL for sessions.
- `gen_mcp2` on hex vs vendored path/git dep (leaning hex: several repos will
  migrate).

## Effort notes (from codebase review, 2026-07-15)

Rough shape of the work: `V2511` entities regeneration (small), legacy
transport + shared SSE extraction (medium, much salvageable from `main`),
session controller + token default (small once data-only), request/result
translation 2025 ↔ core contract (the fiddly heart), `gen_mcp2` release
pipeline (small, mostly scripting + audit). The old stateful test suite on
`main` (~3k lines) is re-targetable for the compat transport.
