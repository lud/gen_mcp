# Hard fork to MCP 2026-07-28 stateless core

Move `gen_mcp` to the MCP `2026-07-28` release candidate as a **hard fork**: the
stateless protocol core becomes the foundation, and the current stateful /
session-based machinery is removed rather than kept alongside.

RC announcement: https://blog.modelcontextprotocol.io/posts/2026-07-28-release-candidate/

## Official documentation (markdown version)

- [Spec Home](https://modelcontextprotocol.io/specification/draft.md)
- [Streamable HTTP Transport](https://modelcontextprotocol.io/specification/draft/basic/transports/streamable-http.md)

Other pages are discoverable from the Spec Home. Agents are welcome to add useful links here for future reference.

## Why hard fork (not dual-stack)

We considered supporting both the stateful (`2025-11-25` / `2025-06-18`) and
stateless (`2026-07-28`) protocols from one codebase. Rejected because, for this
project:

- **No active feature work on the old version** -- it becomes frozen, so there
  is nothing to amortize across a dual codebase. Dual-stack would be pure tax.
- **`0.x` library** -- breaking changes are semver-free; no stability-sensitive
  installed base to protect mid-transition.
- **Timeline lands around the new release** -- no real window where "old clients
  in the wild + new spec not ready" is the problem dual-stack solves.
- **The session machinery is the spine today and pure cost under the new spec.**
  A hard fork lets us *delete* it (`Mux.Session`, `SessionController`, `syn`
  registry, `session_*` callbacks, `Mcp-Session-Id`), making the library
  smaller and simpler -- which matters most under limited bandwidth.

The TypeScript SDK took this same hard-fork path (main = v2 rewrite, `v1.x`
maintenance branch). The Go SDK chose dual-stack; we are deliberately not.

## What survives the fork

This is **not** a from-scratch rewrite. Reused largely unchanged:

- the **codegen pipeline** (`tools/gen-rpc-schemas.exs` -> entities);
- the **Suite component model** (tools / resources / prompts), `Suite.Tool`,
  repos;
- the **`GenMCP.Mux.Channel`** abstraction (reply path);
- **JSV** validation (already JSON Schema 2020-12).

What gets deleted is the session/transport-coupling layer. So: amputate the
spine, re-attach the transport to a stateless dispatch.

## Non-goals (explicitly out of scope)

- **Re-introducing stateful sessions.** The new architecture *could* support an
  optional session layer on top of the stateless core (SEP-2567 explicit state
  handles), but we are **not planning it** -- we may never need it. If a real
  need appears, it is a future feature, not part of this fork.
- **Dual-version / dual-stack support** in one tree.
- **MCP Apps**, auth hardening beyond what already exists -- deferred (see 006 /
  future work).

## Key facts shaping the design (verified 2026-05-29)

- **Schema source: `schema/draft/schema.json`** on `main` of
  `modelcontextprotocol/modelcontextprotocol` (JSON Schema 2020-12, internal
  `DRAFT-2026-v1`). No dated `schema/2026-07-28/` dir exists yet -- the RC lives
  in `draft/` and is a **moving target**; pin a commit SHA for codegen.
- **No `initialize` handshake, no `Mcp-Session-Id`.** Protocol version is the
  authoritative `MCP-Protocol-Version` HTTP header on HTTP transport (SEP-1442);
  client info / capabilities ride in `_meta` under reverse-DNS keys
  `io.modelcontextprotocol/clientInfo`, `/clientCapabilities` (and
  `/protocolVersion`). Modeled as `MetaObject` / `RequestMetaObject`.
- **`server/discover`** (`DiscoverRequest` / `DiscoverResult`) replaces
  `initialize` for capability discovery.
- **`notifications/initialized` is an accept-and-ignore no-op** (SEP-1442) for
  transitional clients.
- **Multi round-trip**: `InputRequiredResult` + `Input*` family (SEP-2322)
  replaces server-initiated requests holding SSE open.
- **Tasks are NOT in core** -- moved to the `io.modelcontextprotocol/tasks`
  extension (separate schema/repo, SEP-2663). `tasks/list` removed.
- Error: missing-resource moves from `-32002` to standard `-32602`.

## Phases (specs)

- **002 Foundation** -- vendor `schema/draft` (SHA-pinned), version the
  generated vocabulary as `GenMCP.MCP.V2607` with a `GenMCP.MCP` facade,
  centralize the protocol version constant.
- **003 Stateless transport + remove session machinery** -- rework
  `StreamableHTTP` to stateless dispatch; delete `Mux.Session` /
  `SessionController` / `syn` / `Mcp-Session-Id`.
- **004 Stateless server contract** -- `GenMCP.Suite` builds state per-request
  from `_meta`; add `server/discover`; drop `session_*` callbacks from the
  `GenMCP` behaviour.
- **005 New 2026 semantics** -- error codes, `InputRequiredResult`,
  `initialized` no-op, Tasks via external extension, cache hints.
- **006 Release packaging + migration docs** -- the client-dependent packaging
  decision, maintenance branch, CHANGELOG / migration + protocol guide.

## Open question (does NOT block 002-005)

Will any downstream client platform send **different-version requests to the
same server endpoint**? Integration feedback pending (asked 2026-05-29).

- If **no** (each server speaks one negotiated version -- the likely case):
  single `gen_mcp` package, `main` goes stateless, cut a `0.10.x` maintenance
  branch.
- If **yes** (one running server must field both): Mix/Hex resolve one version
  per package name, so coexistence requires a **separate package** (the old spec
  frozen as a legacy package; `gen_mcp` evolves). Adopt only if confirmed.

This is a **packaging/release** decision, not an architecture one -- it lives in
spec 006 and does not gate the build work in 002-005.

## Versioning / namespacing

Generate the vocabulary into `GenMCP.MCP.V2607` from the start, with
`GenMCP.MCP` as a thin facade to the latest. Cheap forward-proofing: the next
spec bump is additive (add `V27xx`, flip the facade) instead of a rename. The
generator already parameterizes `base_module/0`.
