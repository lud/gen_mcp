# Stateless server contract

Rework `GenMCP.Suite` and the `GenMCP` behaviour so a server builds its state
per-request from `_meta` instead of from a handshake held in a session process.

## Inherited from spec 003 (re-scoped 2026-06-10)

These 003 items land here because the old Suite is their only remaining
consumer and it is rewritten wholesale in this spec:

- **Delete `lib/gen_mcp/suite/session_controller.ex` + `session_controller/`**
  (Noop, DevSessionStore) and **`lib/gen_mcp/suite/persisted_client_info.ex`** â€”
  wired through `suite.ex` (opts schema `session_controller`, `fetch`/`restore`,
  the `InitializedNotification` clause).
- **Channel breaking drops**: remove `assigns`/`assign/3` (consumers:
  `extension.ex:37`, `tool.ex` docs â€” read-only request context now lives in
  the `meta` field added by 003) and `status`/`set_streaming`/`as_closed` + the
  `send_*` `:closed` guards (consumers: `suite.ex:198,495,550`). Stream-state is
  tracked by the worker wrapper, content-type by the relay, "closed" by the
  relay `:DOWN`.

### Callback surface (decided 2026-06-08)

Every return either **terminates** the request or **continues** it as a stream.
State rides **only** on the continue return (`{:stream, state}`) â€” terminal
returns end the worker, so their state would be discarded and is not carried.
`handle_request` and `handle_message` share one vocabulary.

| Callback | Returns |
| --- | --- |
| `init/1` | `{:ok, state}` \| `{:stop, term}` |
| `handle_request/3` | `{:result, result}` \| `{:error, reason}` \| `{:stream, state}` |
| `handle_notification/3` | `:ok` |
| `handle_message/3` | `{:stream, state}` \| `{:result, result}` \| `{:stop, reason}` \| `{:error, reason}` |

- **`init/1`** â€” the `session_id` argument is gone. Receives the validated server
  opts; per-request data arrives via the request + channel, not init. Runs per
  request (cheap; opts are pre-validated at plug `init`).
- **State on continue only (Q1).** The old `{:reply, server_reply, state}` /
  `{:stop, â€¦, state}` wrapping is gone. With sessions gone there is **no
  long-lived server to stop**, and a terminal return ends the worker â€” so
  `{:result, result}` / `{:error, reason}` carry **no state**. Only `{:stream,
  state}` does, because only it has a successor callback.
- **Unified `:stream` (Q3).** `{:noreply, state}` is folded into `{:stream,
  state}`: both mean "keep the stream open, hold this state, call me on the next
  message." So `handle_request` opting into streaming and `handle_message`
  continuing it use the *same* tag. One model: **terminate or keep streaming.**
- **`handle_message/3`** is the rename of `handle_info/2`. `handle_info` was wrong
  on two counts: a message can now *produce the request's result* (GenServer's
  `:noreply`/`:stop` can't express that), and it implies a generic OTP loop. It is
  invoked **only** after `handle_request` returned `{:stream, state}` (spec 003's
  two-path model). `{:stop, reason}` means *end this stream with no result* (e.g. a
  listener's normal exit) â€” distinct from the deleted server-level stop.
- **Channel is input-only (Q2).** The channel is passed **in** to every callback
  (for `send_progress`/`send_log`) but **never returned** â€” it is immutable
  per-request framework context, not handler state. This is why no return tag
  carries a channel. The mutable `assigns` scratchpad is **removed**; handler data
  lives in `state`, read-only request context in the channel's `meta` (spec 003,
  *Channel becomes inert*). For 007 this means continuation state is passed
  **explicitly** as `{:input_required, requests, kept, state}` (the `kept` blob the
  framework encrypts) rather than auto-packed from assigns.
- **`handle_notification/3`** returns `:ok`: a notification POST gets a `202` with
  no body and never streams, so there is nothing to **return** (its state would be
  discarded too). It still takes the channel **in** â€” like every other callback â€”
  so the handler can read per-request context from `channel.meta` (`client_info`,
  `client_capabilities`, `protocol_version`) plus the merged conn assigns. This is
  the trust/authorization surface for notifications: without it a handler cannot
  tell who sent the notification or under what auth. **Changed 2026-06-15** from
  the earlier `/2` (no channel): the "nothing to return" rationale was about the
  *return value* and was wrongly extended to drop the *input* channel. The channel
  is per-request context that the worker already builds, so passing it is free and
  consistent. See **Notifications** below for the full model.

## Notifications (decided 2026-06-15)

Grounded in the `2026-07-28` draft (`basic/patterns/cancellation`,
`basic/transports/streamable-http`). The spec is explicit that the **stateless
HTTP binding handles cancellation by closing the request's response stream**, not
by a JSON-RPC notification:

> **Streamable HTTP**: Closing the SSE response stream is the cancellation signal.
> The server **MUST** treat a client disconnect as cancellation of that request.
> No `notifications/cancelled` message is required or expected.
> **stdio**: ... the client **MUST** send a `notifications/cancelled`.

So `notifications/cancelled` is the **stdio** mechanism. On the Streamable HTTP
transport this library targets, cancellation **is** connection-close â€” already
wired by spec 003's relay (`spawn_monitor` â†’ `:DOWN`/`:CHAN_DOWN` tears the worker
down). No `requestId â†’ worker` correlation is needed, and none is possible: with
sessions and the `syn` registry deleted (003), there is no lookup table, and the
spec permits ignoring an uncorrelatable notification (*"Receivers **MAY** ignore
... if the referenced request is unknown"*).

### What a stateless HTTP server can receive

Each clientâ†’server notification is its own HTTP POST â†’ `202 Accepted`, never
streams. Disposition under the stateless core:

| Notification | Disposition |
| --- | --- |
| `notifications/initialized` | Transitional no-op, accept-and-ignore (SEP-1442). The transport short-circuits it before validation/dispatch â€” the server is never invoked. Detail in **spec 005**. |
| `notifications/cancelled` | stdio-only mechanism; "not required or expected" on HTTP. Statelessly uncorrelatable (no registry). Accept (202) + ignore. |
| `notifications/progress` | Targets a *server-initiated* request â€” those no longer exist (MRTR / `InputRequiredResult`). Inert. |
| `notifications/roots/list_changed` | Meaningless statelessly â€” nothing holds roots, no subscription. Inert. |

Net: beyond the `initialized` no-op, **no clientâ†’server notification carries a
stateless action**. The `GenMCP` behaviour callback `handle_notification/3` is
itself the optional hook â€” a custom server overrides it to observe/authorize
notifications (using `channel.meta` for trust); the Suite's default is
accept-and-ignore (`:ok`). No SessionController-shaped module is reintroduced.

### Cancellation does *not* travel via the channel

The channel passed to `handle_notification/3` is the **notification's own**
per-request context â€” a sibling of any in-flight request's channel, not a handle
to it. It enables **trust** (read `client_info`/auth from `_meta`); it does **not**
let a notification reach or cancel another in-flight request. Cross-request reach
would require the registry the fork deleted (a feature non-goal). Cancellation
stays transport-level (connection-close).

## Lazy provider resolution (no eager expansion)

Today `initialize_from/3` calls `refresh_extensions(state, channel, :all)` once
at session init, invoking **every** extension's `tools/2` / `resources/2` /
`prompts/2` callback and expanding the full catalog into `%State{}`. Under the
stateless core there is no init to amortize that against â€” it would re-run on
every request (even a `ping` or a single `tools/call`). Replace it with
per-operation lazy resolution.

Cost map (verified in code):

- `Tool.expand/1` is **cheap** â€” `Code.ensure_loaded` + `info(:name)`, returns
  `%{name, mod, arg}`. Safe to run while scanning for a match.
- `Tool.describe/1` (schema `normalize_schema`/JSV build) is **expensive** and is
  **already lazy** (only on `tools/list`). Same for `read`/`get`/`call`.
- The genuinely un-lazy cost is the **per-extension provider callbacks**
  (`Extension.tools/resources/prompts`) â€” user code that may do I/O/auth/DB.
  Laziness exists to avoid invoking extensions we don't need.

Resolution per operation (each stateless request performs exactly one, so there
is no cross-operation reuse to cache):

- **`tools/list`** â€” walk all extensions (listing is inherently total); `expand`
  + `describe` each. No coverage change.
- **`tools/call` name=X** â€” a lazy `Stream` over extensions â†’ `Extension.tools` â†’
  `Tool.expand` (cheap, yields name) â†’ **stop at first name match**, skipping
  every later extension's callback. Only the matched tool is `call`ed.
- **`resources/read` uri / `prompts/get` name** â€” same shape but routed by
  **prefix** (`find_repo` â†’ `String.starts_with?`). Lazily expand repos until a
  prefix matches, then `read`/`get`. `resources/list` / `prompts/list` stay total
  (their existing repo-by-repo pagination is unchanged); `list_resource_templates`
  stays total (needs every repo's template).

### Precedence: first-match-wins (DECIDED)

Resolution stops at the **first** matching name/prefix. `SelfExtension` (the
config `:tools`/`:resources`/`:prompts`) is prepended, so **explicit config wins
over extensions** on a name collision. This **flips today's `Map.new` last-wins
semantics** (where a later extension overrode config) â€” call it out in the
migration notes (spec 006). No need to scan past the first match.

This applies to **both** the lazy single-target path *and* the full-expansion
paths (`tools/list`, `server/discover`). The expansion paths must dedup
**first-wins** (e.g. `Enum.uniq_by(&.name)` over the ordered list, *not*
`Map.new` which keeps the last). All three paths must advertise/resolve the same
winner â€” otherwise a clash could be *called* as config but *listed/advertised* as
the extension's tool.

### Pagination cursors (no per-session key)

`sign_token/2` currently keys cursors with per-session `token_key` + `session_id`
salt â€” neither exists statelessly. Cursors must become self-contained tokens
signed with a **server-wide** key. This is the same key-management problem as
`requestState` (spec 007) â€” **share one keyring/secret mechanism** across cursor
signing and MRTR state. **Re-scoped 2026-06-10:** the key handling lands now as
**task 012**; `requestState` encryption stays in 007 and reuses it.

**Done 2026-06-15 (task 012).** The `use`-module/`otp_app` proposal was dropped
in favour of `GenMCP.Token` â€” a thin functional wrapper over `Phoenix.Token`
(authenticated encryption via the app's `secret_key_base`, read from the Phoenix
endpoint copied onto `Channel.endpoint`). Cursors are **encrypted, not signed**,
purpose-framed by MCP method, namespaced under a `gen_mcp` salt, 20-minute default
TTL. See task 012 for the full design; `token_test.exs` pins the API.

## Decisions (2026-06-10)

- **Tool contract change â€” `{:async, {tag, ref}}` does not survive.** The
  per-request worker owns its process: a slow tool just blocks in `call/3`
  while the relay pumps the socket, so spawning a task to "go async" is
  pointless. A tool that genuinely needs to receive messages aligns with the
  server vocabulary instead:
  - `call/3` returns `{:result, result}` | `{:stream, tool_state}` |
    `{:error, reason}` (no more channel in returns â€” tools were using the
    returned channel's assigns as state; `tool_state` replaces that).
  - `continue({tag, result}, channel, arg)` becomes
    **`continue(message, channel, tool_state, arg)`** (arg always last): the
    Suite forwards **every** worker message of a streaming tool's request to
    it, so user-spawned tasks/subscriptions deliver naturally. `{:stop, _}` is
    deliberately **not** in the tool vocabulary: a `tools/call` must terminate
    with a result or an error (the tests do not cover `{:stop}` for tools).
  - `Tool` is a Suite-private behaviour but it is mocked in tests, so the
    rewritten Suite tests drive this change.
- **`Channel.meta` loosens its typing** (drop the strict map type or use
  `optional(atom) => term`): the transport's `assigns`/`copy_assigns` plug
  options survive and are **merged into `meta`**, which is the single
  read-only request-context surface. The reserved keys extracted from `_meta`
  (`client_info`, `client_capabilities`, `protocol_version`) should be merged
  **after** the conn assigns so a user assign cannot shadow them.

## Stateless runner â€” dropped (resolved 2026-06-15)

The earlier plan for a new `lib/gen_mcp/stateless.ex` glue module is **dropped**:
no such module is needed. `lib/gen_mcp/server.ex` is already the per-request
worker and runs the full `GenMCP` lifecycle â€” `init/1` (line 94) â†’
`handle_request/3` (127) â†’ `handle_notification/3` (144) / `handle_message/3`
(163) â€” building state via `build_state/3` from the request `_meta` and owning the
receive loop for the streaming path. No session GenServer, no registry, no
timeout. The transport (spec 003) spawns this worker directly. Nothing else fills
a separate "runner" role, so the deliverable is struck.

## Acceptance criteria

- Full Suite feature parity over the stateless path.
- No eager full-catalog expansion on the hot path: single-target operations only
  invoke providers up to the first match; `*/list` remain total. `server/discover`
  is the sole exception and may expand the full catalog.
- `server/discover` returns self-describing capabilities (via `capabilities/1`
  over a full expansion), with no user-declared capability config required.
- `GenMCP` behaviour has no session callbacks; `mix check` + dialyzer green.
