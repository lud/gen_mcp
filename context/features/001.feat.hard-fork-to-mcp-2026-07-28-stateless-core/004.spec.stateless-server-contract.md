# Stateless server contract

Rework `GenMCP.Suite` and the `GenMCP` behaviour so a server builds its state
per-request from `_meta` instead of from a handshake held in a session process.

## Inherited from spec 003 (re-scoped 2026-06-10)

These 003 items land here because the old Suite is their only remaining
consumer and it is rewritten wholesale in this spec:

- **Delete `lib/gen_mcp/suite/session_controller.ex` + `session_controller/`**
  (Noop, DevSessionStore) and **`lib/gen_mcp/suite/persisted_client_info.ex`** —
  wired through `suite.ex` (opts schema `session_controller`, `fetch`/`restore`,
  the `InitializedNotification` clause).
- **Channel breaking drops**: remove `assigns`/`assign/3` (consumers:
  `extension.ex:37`, `tool.ex` docs — read-only request context now lives in
  the `meta` field added by 003) and `status`/`set_streaming`/`as_closed` + the
  `send_*` `:closed` guards (consumers: `suite.ex:198,495,550`). Stream-state is
  tracked by the worker wrapper, content-type by the relay, "closed" by the
  relay `:DOWN`.

## `GenMCP` behaviour (`lib/gen_mcp.ex`)

**Drop** the session-oriented callbacks: `session_fetch/3`, `session_restore/3`,
`session_delete/1`, `session_timeout/1` (lines 138-179) and their
`@optional_callbacks`. Also drop `@type session_id` and the duplicate
`server_reply` / `server_reply_nostream` types (lines 120-121, defined
identically).

### Callback surface (decided 2026-06-08)

Every return either **terminates** the request or **continues** it as a stream.
State rides **only** on the continue return (`{:stream, state}`) — terminal
returns end the worker, so their state would be discarded and is not carried.
`handle_request` and `handle_message` share one vocabulary.

| Callback | Returns |
| --- | --- |
| `init/1` | `{:ok, state}` \| `{:stop, term}` |
| `handle_request/3` | `{:result, result}` \| `{:error, reason}` \| `{:stream, state}` |
| `handle_notification/2` | `:ok` |
| `handle_message/3` | `{:stream, state}` \| `{:result, result}` \| `{:stop, reason}` \| `{:error, reason}` |

- **`init/1`** — the `session_id` argument is gone. Receives the validated server
  opts; per-request data arrives via the request + channel, not init. Runs per
  request (cheap; opts are pre-validated at plug `init`).
- **State on continue only (Q1).** The old `{:reply, server_reply, state}` /
  `{:stop, …, state}` wrapping is gone. With sessions gone there is **no
  long-lived server to stop**, and a terminal return ends the worker — so
  `{:result, result}` / `{:error, reason}` carry **no state**. Only `{:stream,
  state}` does, because only it has a successor callback.
- **Unified `:stream` (Q3).** `{:noreply, state}` is folded into `{:stream,
  state}`: both mean "keep the stream open, hold this state, call me on the next
  message." So `handle_request` opting into streaming and `handle_message`
  continuing it use the *same* tag. One model: **terminate or keep streaming.**
- **`handle_message/3`** is the rename of `handle_info/2`. `handle_info` was wrong
  on two counts: a message can now *produce the request's result* (GenServer's
  `:noreply`/`:stop` can't express that), and it implies a generic OTP loop. It is
  invoked **only** after `handle_request` returned `{:stream, state}` (spec 003's
  two-path model). `{:stop, reason}` means *end this stream with no result* (e.g. a
  listener's normal exit) — distinct from the deleted server-level stop.
- **Channel is input-only (Q2).** The channel is passed **in** to every callback
  (for `send_progress`/`send_log`) but **never returned** — it is immutable
  per-request framework context, not handler state. This is why no return tag
  carries a channel. The mutable `assigns` scratchpad is **removed**; handler data
  lives in `state`, read-only request context in the channel's `meta` (spec 003,
  *Channel becomes inert*). For 007 this means continuation state is passed
  **explicitly** as `{:input_required, requests, kept, state}` (the `kept` blob the
  framework encrypts) rather than auto-packed from assigns.
- **`handle_notification/2`** returns `:ok`: a notification POST gets a `202` with
  no body and never streams, so there is nothing to return (its state would be
  discarded too).

## `GenMCP.Suite` (`lib/gen_mcp/suite.ex`)

- Remove the `{:__init__, ...}` two-phase init and the `InitializeRequest`
  handler (lines 122-160) -- there is no handshake.
- Refactor `initialize_from_req/4` (line 572) into
  `build_state(client_info, client_capabilities, opts)` driven by the request
  `_meta`, not an `InitializeRequest` struct.
- The `do_handle_request/3` clauses (lines 166-330) keep their request shapes but
  **stop reading precomputed catalogs** off `%State{}` (`tools_map`,
  `tool_names`, `*_prefixes`, `*_repos`). They resolve providers lazily instead
  — see "Lazy provider resolution" below.
- Add a **`server/discover`** clause:
  `do_handle_request(%MCP.V2607.DiscoverRequest{}, _, state)` returning a
  `DiscoverResult`. **Decision:** `server/discover` **may expand the full
  catalog** (run all extension providers, like the old
  `refresh_extensions(:all)`) and reuse `capabilities/1` (line 704) as-is —
  `tools: map_size(tools_map) > 0` etc. Discover is an **infrequent** call, so
  paying the full-expansion cost here is acceptable and keeps capabilities
  **self-describing** (no requirement for the user to hand-declare them). This is
  the only request that expands eagerly; every other op stays lazy.
- `%State{}` loses `session_id`, the `sc_*` (session controller) fields, **and
  the precomputed catalogs** (`tool_names`, `tools_map`, `resource_prefixes`,
  `resource_repos`, `prompt_prefixes`, `prompt_repos`). What remains is the
  cheap expanded **extension list** plus client info/capabilities/log level.
  Audit `State` (lines 74-100) and trim.

## Lazy provider resolution (no eager expansion)

Today `initialize_from/3` calls `refresh_extensions(state, channel, :all)` once
at session init, invoking **every** extension's `tools/2` / `resources/2` /
`prompts/2` callback and expanding the full catalog into `%State{}`. Under the
stateless core there is no init to amortize that against — it would re-run on
every request (even a `ping` or a single `tools/call`). Replace it with
per-operation lazy resolution.

Cost map (verified in code):

- `Tool.expand/1` is **cheap** — `Code.ensure_loaded` + `info(:name)`, returns
  `%{name, mod, arg}`. Safe to run while scanning for a match.
- `Tool.describe/1` (schema `normalize_schema`/JSV build) is **expensive** and is
  **already lazy** (only on `tools/list`). Same for `read`/`get`/`call`.
- The genuinely un-lazy cost is the **per-extension provider callbacks**
  (`Extension.tools/resources/prompts`) — user code that may do I/O/auth/DB.
  Laziness exists to avoid invoking extensions we don't need.

Resolution per operation (each stateless request performs exactly one, so there
is no cross-operation reuse to cache):

- **`tools/list`** — walk all extensions (listing is inherently total); `expand`
  + `describe` each. No coverage change.
- **`tools/call` name=X** — a lazy `Stream` over extensions → `Extension.tools` →
  `Tool.expand` (cheap, yields name) → **stop at first name match**, skipping
  every later extension's callback. Only the matched tool is `call`ed.
- **`resources/read` uri / `prompts/get` name** — same shape but routed by
  **prefix** (`find_repo` → `String.starts_with?`). Lazily expand repos until a
  prefix matches, then `read`/`get`. `resources/list` / `prompts/list` stay total
  (their existing repo-by-repo pagination is unchanged); `list_resource_templates`
  stays total (needs every repo's template).

### Precedence: first-match-wins (DECIDED)

Resolution stops at the **first** matching name/prefix. `SelfExtension` (the
config `:tools`/`:resources`/`:prompts`) is prepended, so **explicit config wins
over extensions** on a name collision. This **flips today's `Map.new` last-wins
semantics** (where a later extension overrode config) — call it out in the
migration notes (spec 006). No need to scan past the first match.

This applies to **both** the lazy single-target path *and* the full-expansion
paths (`tools/list`, `server/discover`). The expansion paths must dedup
**first-wins** (e.g. `Enum.uniq_by(&.name)` over the ordered list, *not*
`Map.new` which keeps the last). All three paths must advertise/resolve the same
winner — otherwise a clash could be *called* as config but *listed/advertised* as
the extension's tool.

### Pagination cursors (no per-session key)

`sign_token/2` currently keys cursors with per-session `token_key` + `session_id`
salt — neither exists statelessly. Cursors must become self-contained tokens
signed with a **server-wide** key. This is the same key-management problem as
`requestState` (spec 007) — **share one keyring/secret mechanism** across cursor
signing and MRTR state, decided in 007.

## Stateless runner (`lib/gen_mcp/stateless.ex`, new)

Thin glue between the transport (spec 003) and `Suite.handle_request/3`: given
pre-validated opts + the request `_meta` + the channel, build state via
`build_state/3` and run the `GenMCP` lifecycle. It runs **inside the per-request
worker** spawned by the transport (spec 003), which owns the receive loop for the
streaming path (`{:stream, channel, state}` → `handle_message/3`). No session
GenServer, no registry, no timeout.

## Tests

- Suite stateless unit tests: `tools/list`, `tools/call`, `resources/list`
  (pagination), `resources/read`, `prompts/list`, `prompts/get`,
  `server/discover` all work with state built from `_meta`.
- **Laziness:** `tools/call` for a tool in an early extension does **not** invoke
  a later extension's `tools/2` callback (assert via a spy/side-effecting
  extension). Same for `resources/read` / `prompts/get` prefix routing.
- **Precedence (all paths must agree):** a name defined in both config and an
  extension resolves to the **config** one (first-match-wins) on **every** path,
  not just `tools/call`:
  - `tools/call` — invokes/`describe`s the config tool.
  - `tools/list` — lists the config tool **once** (no duplicate), with the
    config tool's metadata, not the extension's.
  - `server/discover` — the full-expansion path must dedup **first-wins** too, so
    the advertised catalog matches what `tools/call`/`tools/list` resolve.
  - Guard against the current `Map.new/1` **last-wins** dedup: the test must fail
    if discover/list advertise the extension's clashing tool while call resolves
    the config one.
- Confirm no `session_*` callbacks remain in the behaviour (compile-time:
  modules implementing `GenMCP` no longer need them).

## Acceptance criteria

- Full Suite feature parity over the stateless path.
- No eager full-catalog expansion on the hot path: single-target operations only
  invoke providers up to the first match; `*/list` remain total. `server/discover`
  is the sole exception and may expand the full catalog.
- `server/discover` returns self-describing capabilities (via `capabilities/1`
  over a full expansion), with no user-declared capability config required.
- `GenMCP` behaviour has no session callbacks; `mix check` + dialyzer green.
