# MRTR: multi round-trip + `requestState`

> **STATUS: DONE.** Design resolved and implemented; the full test plan (see
> "Test plan / status") is green. Continuation state is encrypted with
> `GenMCP.Token` (`{:reqstate, map}` purpose, no key rotation); MRTR is delivered
> as a plain `{:result, %InputRequiredResult{}}` (Design B ‚Äî runner stays
> generic), with all encryption / binding / `resultType` / error mapping owned by
> `GenMCP.Suite`. Split out of spec 005 because it is a whole subsystem (crypto +
> dispatch resume), not a one-off semantic.

## What MRTR is (SEP-2322)

Spec source: `docs/specification/draft/basic/patterns/mrtr.mdx` in the
`modelcontextprotocol/modelcontextprotocol` GitHub repo (the `.mdx` the rendered
page is built from; verified present 2026-06-17), rendered at
<https://modelcontextprotocol.io/specification/draft/basic/patterns/mrtr>.
**Note:** the page moved from `basic/utilities/mrtr` to `basic/patterns/mrtr`;
the old path 404s.

In the stateless core, the server can **no longer send JSON-RPC requests** to the
client (no held-open SSE stream for sampling/elicitation/roots). Instead:

1. Client sends an initial request (e.g. `tools/call`).
2. Server needs more info; returns an **`InputRequiredResult`** carrying
   `inputRequests` (elicitation / sampling / list-roots asks) **and an opaque
   `requestState` blob**. The initial request is **terminated** here.
3. Client gathers the answers and **retries the original request** ‚Äî a brand new
   JSON-RPC request with a **different `id`** ‚Äî echoing `requestState` plus
   `inputResponses`.
4. Server decrypts `requestState`, rehydrates, merges `inputResponses`, resumes.

Key spec invariant (verbatim): *"the requests in each step are completely
independent: the server processing the retry does not need any information beyond
what is directly present in the retry request."* Any node behind a load balancer
must be able to field the retry ‚Äî so **all continuation state lives inside the
encrypted `requestState`**, never on the server. The client likewise MUST echo
`requestState` verbatim and MUST NOT inspect or reuse it on any parallel request.

## Layering: framework vs Suite (resolves open Q3)

`requestState` touches three layers; keep their responsibilities disjoint. The
key decision: **the runner stays generic ‚Äî MRTR is "just a result", not a runner
concept.** `InputRequiredResult` flows through the existing `{:result, result}`
path with no special-casing, so `GenMCP.Server` needs no MRTR knowledge and no
coupling to the `V2607` vocabulary.

- **Runner / `GenMCP` behaviour (spec 004) ‚Äî unchanged.** It forwards whatever
  `result` a handler returns. `InputRequiredResult` is added to the `GenMCP`
  `result` type so a custom server's `{:result, %InputRequiredResult{}}`
  type-checks, but there is **no** `:input_required` return variant at the
  behaviour level and the runner does not build, inspect, or encrypt anything.
  A retry is just another request carrying a `requestState` field; the handler
  decides what to do with it.
- **`GenMCP.Token` ‚Äî framework-level helper, NOT Suite-scoped.** Already exists
  for pagination cursors; this spec adds the `{:reqstate, ‚Ä¶}` purpose. Any layer
  may use it; Suite is one consumer.
- **`GenMCP.Suite` (this spec's policy home) ‚Äî owns the whole MRTR mechanism.** A
  Suite *tool* returns the convenience `{:input_required, requests, client_state}`
  tuple; `GenMCP.Suite.Tool` transforms it into `{:result, %InputRequiredResult{}}`,
  encrypting `client_state` into `requestState` via `GenMCP.Token` with a
  tool+args binding. On a retry it decrypts (binding to the retry's tool+args),
  resumes via `call/3`, and maps decrypt failures to JSON-RPC errors. The
  encryption, binding, `resultType` stamping, and error mapping all live here.
- **Custom servers** implementing `GenMCP` directly build the
  `%InputRequiredResult{}` themselves and return `{:result, ‚Ä¶}`, owning their own
  `requestState` bytes (a public `MCP.input_required_result/2` builder stamps
  `resultType`; integrity protection is their choice ‚Äî `GenMCP.Token` is available
  but not required).

## Behaviour / Suite surface

- **No new `GenMCP` behaviour return.** MRTR is delivered as an ordinary
  `{:result, %InputRequiredResult{}}`. The only behaviour-level change is adding
  `MCP.InputRequiredResult.t()` to the `GenMCP` `result` type (done).
- **`GenMCP.Suite.Tool` gains the convenience variant.** Its `call_result` type
  accepts `{:input_required, requests, client_state}` (3-tuple ‚Äî no channel,
  tools no longer return one); the dispatch in `Tool` encrypts + wraps it into a
  `{:result, %InputRequiredResult{}}`. `client_state` is a plain term the tool
  author hands over without touching crypto or the MCP struct.
- **Three-tuple, not four.** Earlier drafts carried `{:input_required, requests,
  kept, state}`. There is no separate live `state`: the request is terminal
  here. (Contrast `:stream`, which keeps the process spinning and *does* thread a
  live handler `state`.) The single continuation blob is `client_state`.
- On a retry request carrying `requestState` + `inputResponses`: the Suite
  decrypts the blob (bound to the retry's tool+args), swaps the decrypted
  `client_state` term into `req.params.requestState` (in place of the wire blob),
  and re-invokes `call/3`. The channel is inert, read-only request context (spec
  003/004) ‚Äî no assigns to rehydrate.
- **Generated shapes (verified against `GenMCP.MCP.V2607`, spec 002):**
  `inputRequests` and `inputResponses` are **maps keyed by server-assigned ids**
  (objects with `additionalProperties`), **not lists** ‚Äî so `requests` in the
  tool's return tuple is a `%{id => InputRequest}` map, and the retry's
  `inputResponses` come back keyed by the same ids. `requestState` is typed
  `string()` everywhere (`InputRequiredResult`, `CallToolRequestParams`,
  `InputResponseRequestParams`), so `GenMCP.Token`'s URL-safe output drops in
  directly. `InputRequiredResult` requires only `resultType`; "at least one of
  `inputRequests`/`requestState`" is a description-level constraint, not
  JSV-enforced ‚Äî this path must uphold it.
- **`resultType` ownership.** The `GenMCP.MCP.V2607` result builders always set
  `resultType: "complete"` and never accept it as an option (see `TODO(spec 007)`
  / `@result_type_complete` in `lib/gen_mcp/mcp/v2607.ex`). The
  `InputRequiredResult` produced by this path is the **only** producer of
  `resultType: "input-required"`; a dedicated `input_required_result/2` builder
  stamps it once, so no per-result builder ever needs to know about it.

## The `requestState` envelope: `GenMCP.Token` (resolves open Q2)

The spec calls `requestState` an **AEAD-protected blob** and requires the server
to *"protect its integrity (e.g. HMAC or AEAD) and MUST reject state that fails
verification."* We satisfy this with the existing `GenMCP.Token`, a thin layer
over `Phoenix.Token` (XChaCha20-Poly1305 AEAD via `Plug.Crypto`):

- **Key source:** derived from the application's `secret_key_base` (carried on the
  channel's endpoint). A blob minted on one node is valid on any node sharing that
  config ‚Äî exactly the load-balancer portability the spec invariant demands.
- **No key rotation.** State surviving a deployment-wide key change is an explicit
  non-goal; a key change simply invalidates in-flight round-trips (acceptable ‚Äî
  see open item on deploys below). This removes the keyring / key-id-prefix
  machinery earlier drafts agonized over.
- **Expiry:** `GenMCP.Token` embeds `:max_age` (default 20 min) inside the
  encrypted payload and enforces it on decrypt ‚Äî a tamper-proof, mint-time replay
  bound that no decrypt site can forget.

### Binding via the purpose (resolves Q2 ‚Äî "no reuse across tool calls")

`GenMCP.Token` folds its `purpose` into key derivation (the PBKDF2 salt), so a
blob only decrypts under the **exact** purpose it was minted for. Extend the
`purpose` type to accept a **unicity map** for request state:

```elixir
@type purpose :: {:cursor, String.t()} | {:reqstate, map}
```

- `Token` hashes the map internally and deterministically:
  `:erlang.term_to_binary(unicity_params, [:deterministic])` (canonical key
  order) ‚Üí `:crypto.hash(:sha256, ‚Ä¶)` ‚Üí fed into the salt. **This hash is salt
  material / a domain-separation label, not a standalone security primitive** ‚Äî
  `Plug.Crypto` does the real KDF + AEAD. The only property required of it is
  collision-resistance for domain separation, which SHA-256 supplies. (We are not
  hand-rolling crypto.)
- **Suite passes `%{tool: tool_name, args: arguments}`** ‚Äî the tool's
  `arguments` map, **not** the raw params object. This matters: per the generated
  `CallToolRequestParams` (verified, spec 002), the retry is the *same* params
  object with `inputResponses` + `requestState` **added**, so the initial params
  (`%{name, arguments, _meta}`) and the retry params
  (`%{name, arguments, _meta, inputResponses, requestState}`) are **not**
  byte-identical. The stably-echoed business input is `arguments`. `_meta` is
  excluded too ‚Äî it is per-request transport metadata (fresh request id, progress
  token, client info/capabilities) and is not stable across the retry. So the
  digest is over `{tool, arguments}` only: identical on initial and retry, while a
  blob minted for tool A / args X stays cryptographically `:invalid` under tool B
  or args Y. The tool author never thinks about binding.
- **Escape hatch (deferred):** narrowing `arguments` to a subset is a future
  option if a real need appears (non-deterministic or oversized arguments). Not in
  scope now.

### Node portability of the kept term ‚Äî not policed

For a retry to resolve on a *different* node, `client_state` should be
**node-portable plain data**. But the framework does **not** enforce this, and an
earlier draft's "validate that `client_state` is serializable and fail loudly"
plan was based on a wrong premise: `:erlang.term_to_binary/2` encodes `pid` /
`ref` / `port` **without error**, and `binary_to_term` decodes them on any node
(even one never connected to the origin) ‚Äî so there is nothing to detect at the
serialization layer. A kept pid simply decodes into a dangling reference to a
process on the origin node (and since `:input_required` is *terminal*, that
process is already dead ‚Äî even on the same node).

Detecting non-portable terms would require an explicit recursive scan for
`pid`/`ref`/`port`/`fun` on every continuation, which we deliberately skip:

- it adds cost to every round-trip for a narrow developer footgun;
- a kept pid is self-defeatingly useless rather than dangerous;
- on **stable, connected** clusters a developer may *legitimately* keep a pid in
  `client_state` and have it resolve.

So this is **documented guidance, not a guard**: keep `client_state` to plain
data unless your deployment guarantees the referenced resources survive and are
reachable on the resuming node. The developer owns the trade-off.

## Replay semantics: bounded, not single-use

Per the spec's own caveat, TTL + binding *"bound the replay window and prevent
cross-user and cross-request reuse, but do not by themselves guarantee
single-use."* gen_mcp is **stateless**, so it **cannot** enforce at-most-once
without server state ‚Äî and does not try to. Document `requestState` as
**replay-bounded** (tool+params binding + TTL), **not single-use**. This is
spec-blessed for the general case; a server needing one-time redemption must add
its own server-side enforcement (out of scope for the stateless core). The
client-side "MUST NOT reuse on a parallel request" is a *client* obligation we
cannot enforce, but the per-tool binding makes accidental cross-tool reuse fail
closed anyway.

## Error handling for an invalid `requestState` (resolves Q1)

The official spec is **silent on a dedicated error code**: MRTR only mandates that
the server *MUST reject state that fails verification*, and the general guidance
is to *"return a JSON-RPC error response with an appropriate error code."* The
draft's error table defines no `requestState`-specific code (and notably no
`-32002` resource-not-found anymore ‚Äî that moved to `-32602`).

**Decision (Suite-level): reuse the cursor pattern.** `GenMCP.Token.decrypt/4`
already returns `{:error, :invalid}` (forged / tampered / wrong key / wrong
purpose) and `{:error, :expired}`. Suite maps these exactly as it maps cursor
failures today (`lib/gen_mcp/error.ex`, `:invalid_cursor` / `:expired_cursor` ‚Üí
`-32602`):

- `:invalid_request_state` ‚Üí `-32602` (Invalid params), HTTP 200
- `:expired_request_state` ‚Üí `-32602` (Invalid params), HTTP 200

`-32602` is what the spec itself mandates for a malformed required request field
elsewhere, and HTTP 200 + JSON-RPC error matches the existing application-error
convention. A custom server is free to pick its own code.

## Open questions (remaining ‚Äî do not block starting)

1. ~~Opt-out API shape.~~ **Resolved:** moot ‚Äî assigns removed; continuation state
   is the explicit `client_state` term.
2. ~~Key source + rotation + missing-key failure mode.~~ **Resolved:**
   `GenMCP.Token` / `secret_key_base`, no rotation; missing endpoint ‚Üí the
   existing `ArgumentError` from `Token` (a server returning `:input_required`
   without a configured endpoint is a misconfiguration, fail loudly).
3. ~~004/007 ownership boundary.~~ **Resolved (Design B):** the runner
   (`GenMCP.Server`) is untouched ‚Äî MRTR is just a `{:result, %InputRequiredResult{}}`
   on the existing path. 007 (`GenMCP.Suite`) owns the *entire* mechanism: the
   `Tool` convenience tuple ‚Üí result transform, Token encryption, binding,
   `resultType` stamping, and error mapping. `Token` is a shared framework
   helper; custom servers build the `InputRequiredResult` themselves.
4. ~~Size ceiling on `requestState`.~~ **Resolved:** we do **not** add our own
   bound ‚Äî we rely on Phoenix/Plug's built-in request-body ceiling
   (`Plug.Parsers` default `:length`, 8 MB; an oversized body raises
   `Plug.Parsers.RequestTooLargeError` ‚Üí HTTP 413 *before* dispatch). Since
   `requestState` rides inside the request body, this caps it for free. **Guard
   it with a regression test** that POSTs a body exceeding the default to a router
   /endpoint configured **with no `:length` override** (testing Phoenix's
   *defaults*, not our config) and asserts rejection. The test is deliberately
   exercising Phoenix, not our code ‚Äî a comment above it must say so, so that if a
   future Phoenix/Plug release ships with the ceiling disabled by default we are
   alerted rather than silently unbounded.
5. **Deploys mid-round-trip** ‚Äî a `secret_key_base` change or expiry lapse
   invalidates in-flight round-trips. **Acceptable** given the short
   default TTL and stateless non-goals; confirm no client retries beyond TTL in a
   way that needs a clearer error than `:expired_request_state`.
6. ~~Verify generated shapes.~~ **Resolved (2026-06-17):** checked against
   `lib/gen_mcp/mcp/v2607/entities.ex`. `requestState` is `string()` everywhere;
   `inputRequests`/`inputResponses` are maps keyed by server-assigned ids (not
   lists); `InputRequest` = `anyOf [CreateMessageRequest, ListRootsRequest,
   ElicitRequest]`, `InputResponse` = `anyOf [CreateMessageResult, ListRootsResult,
   ElicitResult]`; `CallToolRequestParams` carries `inputResponses` + `requestState`
   added to `name`/`arguments`/`_meta`. **This drove the binding correction above**
   (bind `{tool, arguments}`, not whole params, because the retry params differ
   from the initial by the added envelope fields).

## Acceptance criteria

- A Suite tool can return `{:input_required, requests, client_state}` (emitted as
  a `{:result, %InputRequiredResult{}}`), and a client retry resolves to the final
  result on a *different* server node.
- `requestState` is AEAD-encrypted via `GenMCP.Token`, bound to `{tool, args}`,
  and expires (default 20 min).
- A `requestState` minted for tool A (or args X) is rejected as
  `:invalid_request_state` on tool B (or args Y); an expired one is rejected as
  `:expired_request_state`. Both surface as `-32602`.
- Phoenix's default request-body ceiling is in force (regression-tested against
  defaults), bounding `requestState` size without bespoke logic.
- `client_state` node-portability is **documented guidance, not a guard** ‚Äî the
  framework does not scan for or reject `pid`/`ref`/`port`/`fun` (they serialize
  and round-trip fine; meaningfulness on resume is the developer's call).
- `requestState` is documented as replay-bounded, not single-use.
- Our usage of Phoenix.Token and secret key base must not be an attack vector that would allow to discover the secret key base value

### Resolved ‚Äî tool-side resume: reuse `call/3`, no new callback

No dedicated resume callback and no extra argument. On a retry the Suite
decrypts the blob and **re-invokes `c:call/3` with the same request**, swapping
the encrypted `requestState` wire value for the decrypted `client_state` term in
that field (crypto stays in the Suite; the blob never reaches the tool). The
tool branches on the request itself:

- `req.params.requestState` ‚Äî `nil` on the initial call, its kept `client_state`
  term on a resume;
- `req.params.inputResponses` ‚Äî the client's gathered answers.

Everything the tool needs is on the request; "dig into the request" is the whole
API.
