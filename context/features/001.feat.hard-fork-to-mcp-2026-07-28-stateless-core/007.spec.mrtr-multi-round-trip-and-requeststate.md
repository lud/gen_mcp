# MRTR: multi round-trip + `requestState`

> **STATUS: DRAFT — NOT READY FOR IMPLEMENTATION.** This spec captures the design
> direction and open questions for Multi Round-Trip Requests. Refine the open
> questions below and the crypto/key-management section before writing code.
> Split out of spec 005 because it is a whole subsystem (crypto + delta tracking
> + dispatch resume), not a one-off semantic.

## What MRTR is (SEP-2322)

Spec source: `docs/specification/draft/basic/utilities/mrtr.mdx`
(rendered: <https://modelcontextprotocol.io/specification/draft/basic/utilities/mrtr>).

In the stateless core, the server can **no longer send JSON-RPC requests** to the
client (no held-open SSE stream for sampling/elicitation/roots). Instead:

1. Client sends an initial request (e.g. `tools/call`).
2. Server needs more info; returns an **`InputRequiredResult`** carrying
   `inputRequests` (elicitation / sampling / list-roots asks) **and an opaque
   `requestState` blob**. The initial request is **terminated** here.
3. Client gathers the answers and **retries the original request** — a brand new
   JSON-RPC request with a **different `id`** — echoing `requestState` plus
   `inputResponses`.
4. Server decrypts `requestState`, rehydrates, merges `inputResponses`, resumes.

Key spec invariant (verbatim): *"the requests in each step are completely
independent: the server processing the retry does not need any information beyond
what is directly present in the retry request."* Any node behind a load balancer
must be able to field the retry — so **all continuation state lives inside the
encrypted `requestState`**, never on the server.

## Design: explicit kept-state → `requestState` (revised 2026-06-08)

> **SUPERSEDED.** The assigns-based auto-packing design below is **obsolete**. The
> 2026-06-08 channel rework (spec 003/004) **removes `Channel.assign/3` and the
> mutable `assigns` map entirely** — the channel is now inert, read-only request
> context, and handlers carry their own `state`. There is therefore no assign map
> to delta-track or auto-pack.
>
> **New approach — explicit kept-state.** A handler that needs a round-trip
> returns `{:input_required, input_requests, kept, state}` where `kept` is exactly
> the term the framework encrypts into `requestState`. The handler names what to
> keep; the framework encrypts/decrypts and rehydrates it (handed back on the
> retry). This **deletes** the `base_keys` delta-tracking, the transient-key
> opt-out, and the silent-drop hazard — and resolves open questions #1 (opt-out
> API: moot) below. The serialization contract and the crypto envelope still
> apply, now to `kept` instead of an assigns delta.
>
> The subsections below are retained for history; ignore the assigns mechanics.

### Delta tracking (don't pack baseline assigns)

- Snapshot the assign **key set at channel creation** (`create_new/3`) into a new
  `base_keys` field on the `Channel` struct.
- At pack time, pack only `Map.drop(assigns, base_keys)` — request-added keys.
- Rationale: baseline assigns are reconstructed fresh each request from `_meta` +
  server config; packing them would be wasteful and could go stale. **Baseline
  keys are excluded even if a handler overwrites them** (the override is
  recomputed too).

### Opt-out for transient assigns

Need an API to add an assign that is *not* packed (e.g. a DB conn, a parsed
struct, a `pid`). Open question on the exact shape:

- Option A: `Channel.assign(ch, key, val, transient: true)` flag.
- Option B: a separate `Channel.assign_transient/3` + keep a `transient_keys`
  MapSet alongside `base_keys`; pack = `assigns - base_keys - transient_keys`.
- **Lean: Option B** (explicit, symmetric with `base_keys`, no kwarg on the hot
  path). Confirm during refinement.

### Serialization contract (node portability)

Packed state must be **node-portable** — any instance resumes it. So kept terms
must be plain data: **no `pid` / `ref` / `fun` / port / ETS handle**.

- Pack: `:erlang.term_to_binary(delta)`.
- Unpack: `:erlang.binary_to_term(bin, [:safe])` (guards atom-table exhaustion
  and code-loading on hostile input).
- Validate kept state is serializable; **fail loudly** (raise/log) when a handler
  keeps an unserializable term, rather than silently dropping it.

## Design: the `requestState` envelope (crypto)

The spec calls `requestState` an **AEAD-protected blob**. Proposed construction:

```
requestState = base64( key_id || nonce || AEAD_AES_256_GCM(
    key      = server_secret[key_id],
    nonce    = random 96-bit (NEVER reused),
    aad      = { method, tool_name, protocol_version },   # binds context
    plaintext = term_to_binary({ expiry, handler_id, kept_assigns_delta })
))
```

- **AAD binding** prevents a blob minted for tool A being replayed into tool B,
  or across protocol versions. (`:crypto.crypto_one_time_aead/7`.)
- **Expiry** timestamp inside the plaintext bounds the replay window.
- **`handler_id`** records which handler/tool to resume — the retry is a *new*
  request, so the framework must know what it is continuing (this is the part the
  first sketch did not mention).
- **Key management (OPEN):** where does `server_secret` come from? Config? How is
  rotation handled (key-id prefix → keyring lookup)? What is the failure mode
  when no key is configured but a handler returns `InputRequiredResult`? **Must
  resolve before implementation.**

## Behaviour / runner surface

- Add an `{:input_required, input_requests, kept, state}` variant to the unified
  `handle_request` / `handle_message` return set (`GenMCP` behaviour — see spec
  004's callback table). It is a **terminal** return: the stateless runner (spec
  004) turns it into an `InputRequiredResult` **plain JSON result** (not an SSE
  stream) with `kept` encrypted into `requestState`. `kept` is the explicit
  continuation blob (no assigns auto-pack).
- **Boundary question (OPEN):** does 007 own the `{:input_required, …}` return
  type and packing, or does 004 own the return type and 007 only the
  pack/unpack/crypto? Decide so 004 and 007 do not both touch the runner.
- On an incoming request carrying `requestState` + `inputResponses`: decrypt
  `kept`, merge responses, dispatch to `handler_id`. (No channel assigns to
  rehydrate — the handler receives the decrypted `kept` directly.)
- **`resultType` ownership.** The `GenMCP.MCP.V2607` result builders always set
  `resultType: "complete"` and never accept it as an option (see `TODO(spec 007)`
  / `@result_type_complete` in `lib/gen_mcp/mcp/v2607.ex`). This path is the
  **only** producer of `resultType: "input-required"` — via the
  `InputRequiredResult` it emits for `{:input_required, …}`. So no per-result
  builder ever needs to know about it; keep the choice here, request-driven.

## Open questions (resolve before implementation)

1. ~~Opt-out API shape (A vs B above).~~ **Resolved:** moot — assigns removed,
   continuation state is passed explicitly as `kept` (2026-06-08).
2. Key source + rotation + missing-key failure mode.
3. 004/007 ownership boundary for the `{:input_required, …}` return type.
4. Size ceiling on `requestState` — proxies/clients may cap body size; do we
   bound the kept-state size and error past a threshold?
5. Does `handler_id` need to survive code reloads / deploys mid-round-trip, or is
   a short expiry acceptable (a deploy invalidates in-flight round-trips)?
6. Interaction with the `Input*` entity shapes generated in spec 002 — verify
   the generated `InputRequiredResult` / `inputRequests` / `inputResponses`
   fields match this design.

## Acceptance criteria (provisional)

- A handler can return `{:input_required, requests, kept, state}`, and a client
  retry resolves to the final result on a *different* server node.
- `requestState` is AEAD-encrypted, context-bound, and expires.
- Transient assigns never appear in the blob; baseline assigns never appear in
  the blob.
- Unserializable kept state fails loudly, not silently.
