# Validate `Mcp-Method` / `Mcp-Name` headers against the body

Deferred from spec 003 (decided 2026-05-31: tolerate for now). The
`MCP-Protocol-Version` header/body validation **is** done in 003; this task
covers the other two mirrored headers.

## What the spec requires

From `draft/basic/transports` (Request Metadata → Standard Request Headers +
Server Validation):

- `Mcp-Method` (mirrors body `method`) is **REQUIRED** for all requests and
  notifications.
- `Mcp-Name` (mirrors `params.name` or `params.uri`) is **REQUIRED** for
  `tools/call`, `resources/read`, `prompts/get`.
- A server that processes the body **MUST** reject when a header value does not
  match the body, or a required header is missing/malformed, with
  `400 Bad Request` + JSON-RPC error `-32001` (`HeaderMismatch`).
- Header *names* are case-insensitive; header *values* (method names) are
  case-sensitive.

The point: load balancers/gateways route on the header while the server
executes on the body — they must agree, and doing it once at the transport
means individual `GenMCP` implementations don't re-check it.

## Why deferred / what it costs

Our positive-path test client currently sends only `MCP-Protocol-Version`. To
enforce the above we must make the test client a **conforming client**: derive
`Mcp-Method` from `method`, `Mcp-Name` from `params.name`/`uri`, and keep them
consistent with the body. Until then the transport tolerates their absence.

## Work

- Transport: parse + validate `Mcp-Method` / `Mcp-Name` against the decoded
  body; reject mismatches/missing with `400` + `-32001`.
- Test support (`test/support/client.ex`): mirror the headers from the body on
  every conforming POST so the existing positive tests keep passing.
- Tests: add `-32001` cases for each missing/mismatch scenario (mirrors the
  `MCP-Protocol-Version` tests already in `streamable_http_test.exs`).
- Note: the `x-mcp-header` / `Mcp-Param-*` feature is the same "header
  mirroring" family but a larger, separately-tracked piece of work.
