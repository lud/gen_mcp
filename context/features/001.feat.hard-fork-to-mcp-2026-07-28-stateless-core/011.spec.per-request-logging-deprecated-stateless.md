# Per-request logging (deprecated, stateless)

Rework logging for the 2026-07-28 stateless core. Logging is **deprecated** but
stays supported; the stateful `logging/setLevel` request is gone and the verbosity
is now declared **per request** in `_meta`.

## Verified against the spec (2026-06-08)

Source: <https://modelcontextprotocol.io/specification/draft/server/utilities/logging>
(SEP-2577). Confirmed against the generated `GenMCP.MCP.V2607` vocabulary.

- **Deprecated** as of `2026-07-28`. Under the feature-lifecycle policy it remains
  in the spec for **at least twelve months** before becoming eligible for removal.
  New implementations **SHOULD NOT** adopt it. We keep supporting it for the fork.
- **No `logging/setLevel`.** `SetLevelRequest` / `logging/setLevel` do **not**
  exist in V2607 (they remain only in the old `GenMCP.MCP` vocabulary). There is
  no stateful level to hold — there is no session.
- **Per-request level** via `_meta` **`io.modelcontextprotocol/logLevel`**
  (present in `RequestMetaObject`, typed `LoggingLevel`).
- **Absent level ⇒ disabled (MUST).** Verbatim: *"The server **MUST NOT** emit
  `notifications/message` for a request that does not include this field."*
- When present, the server **MAY** send `notifications/message` at or above the
  requested level **on that request's response stream, before the final
  response**. It is **request-scoped**: the server **MUST NOT** deliver it on a
  `subscriptions/listen` stream (spec 008) or any other stream.
- **Invalid level ⇒ reject `-32602`** (Invalid params). Internal errors `-32603`.
- Servers that emit logs **MUST** declare the `logging` capability (via
  `server/discover`, spec 004).
- Log notifications are still `notifications/message` with `level` / `logger` /
  `data` (`LoggingMessageNotification`).

## Design

### Channel carries the level as read-only request context

Fits the inert-channel model (spec 003/004): the transport extracts
`io.modelcontextprotocol/logLevel` from the request `_meta` into the channel.
**Default is `nil` (disabled)** — there is no library default level anymore.

- Drop `GenMCP.default_channel_log_level/0` (the `:notice` default). Absence is
  not a default level; it is *off*.
- `Channel.from_request/2` reads `io.modelcontextprotocol/logLevel` alongside
  `progressToken`. Unknown/unrecognized value → the transport rejects the request
  with `-32602` (a transport-layer validation, like the protocol-version check in
  spec 003), so the channel only ever holds a valid level or `nil`.

### `Channel.send_log/4` is a no-op when disabled

```
log_level == nil          -> no-op (MUST NOT emit), returns :ok
level <  log_level        -> filtered, returns :ok
level >= log_level        -> emits notifications/message
```

`nil` short-circuits **before** the level comparison. `send_log` stays safe to
call unconditionally from any handler — when the client did not opt in, it simply
does nothing. (`send_progress` is unaffected; progress is gated on
`progressToken`, not the log level.)

## Tests (the point of this spec)

Add to the transport conn-case suite (and/or the Suite unit tests):

- **No-op when absent (primary):** a streaming handler that calls
  `Channel.send_log` for a request whose `_meta` carries **no**
  `io.modelcontextprotocol/logLevel` emits **zero** `notifications/message` — only
  the final result reaches the client. Assert the stream contains the result and
  **no** message notification. This is the MUST.
- **Emits when present:** the same handler, with `_meta`
  `io.modelcontextprotocol/logLevel: "warning"`, emits a `notifications/message`
  for an `:error` log (≥ warning) before the result.
- **Filters below the level:** with level `"error"`, an `:info` log is dropped; a
  `:critical` log passes.
- **Invalid level ⇒ -32602:** a request whose `io.modelcontextprotocol/logLevel`
  is not a recognized level is rejected by the transport with `-32602`; the server
  behaviour is never invoked.
- **Capability:** `server/discover` advertises `logging: {}` when the server can
  emit logs (cross-check in spec 004's discover tests).

Replace the old `logging/setLevel` test — that method no longer exists. Drive the
logging-notification path from a normal request (e.g. a streaming `tools/call`)
with the level set in `_meta`.

## Acceptance criteria

- `Channel.send_log` never emits when the request omitted
  `io.modelcontextprotocol/logLevel`; covered by a failing-without-the-fix test.
- Level is read per-request from `_meta`; no stateful `setLevel`; no library
  default level.
- Invalid level rejected with `-32602` at the transport.
- `mix check` green.
