# Server-wide cursor signing key module

Pagination cursors can no longer be keyed per session (`token_key` +
`session_id` salt are gone). They must be self-contained tokens verifiable by
**any** per-request state, on any node — signed with a server-wide key.

## Current state (verified 2026-06-10)

Cursors are **signed, not encrypted**: `Plug.Crypto.sign/3` over the
`{repo_index, repo_cursor}` tuple. The payload is readable by anyone who
Base64-decodes the token — only tampering is prevented. That is acceptable for
the index, but note that `repo_cursor` is repo-author-controlled and is exposed
verbatim; either document that repos must not put secrets in their cursors, or
switch to `Plug.Crypto.encrypt/4` and not have to think about it.

## Design (decided 2026-06-13)

The earlier `use GenMCP.<TokenModule>, otp_app: ...` proposal is dropped. No
user-defined module, no router/Suite option: Phoenix is already a direct
dependency, `Phoenix.Token.encrypt/decrypt` provides authenticated encryption
(XChaCha20-Poly1305 via `Plug.Crypto.MessageEncryptor`), and it accepts a
Phoenix endpoint as key source (it reads `endpoint.config(:secret_key_base)`).

- **Key source**: the transport copies `conn.private.phoenix_endpoint` into a
  new `endpoint` field on `GenMCP.Mux.Channel`. The server-wide key is the
  app's regular `secret_key_base` — shared across nodes by construction.
- **`GenMCP.Token`** (`lib/gen_mcp/token.ex`, implemented 2026-06-13): a thin
  *functional* wrapper over `Phoenix.Token` (no `use` macro). API:
  `encrypt(key_source, purpose, term, opts)` /
  `decrypt(key_source, purpose, token, opts)` returning
  `{:ok, term} | {:error, :invalid | :expired | :missing}`. `key_source` is a
  `%Channel{}` or anything `Phoenix.Token` accepts (endpoint module, key-base
  string, conn). Its value-add over raw `Phoenix.Token`:
  - **framed purposes** `{kind, qualifier}` where kind is `:cursor` or
    `:reqstate` and the qualifier is free-form: cursors are qualified by the
    MCP method they paginate (`"resources/list"`, `"prompts/list"`,
    `"resources/templates/list"`), request states by the tool name. The
    purpose participates in key derivation, so a `resources/list` cursor
    replayed on `prompts/list` is `:invalid` (a repo must never receive an
    authenticated cursor it did not issue), and no cursor can pass as a
    request state;
  - a **`gen_mcp` salt namespace** — the host app shares the same endpoint /
    `secret_key_base` for its own `Phoenix.Token` tokens; the prefix keeps
    the two worlds non-interchangeable (note: `Phoenix.Token`'s second arg
    is named `secret` but is really a PBKDF2 *salt* over `secret_key_base`);
  - the `%Channel{}` head, so the Suite passes what it already holds; a
    channel without endpoint raises `ArgumentError` (defensive only — in
    practice the transport always sets it, the branch is deliberately
    untested).
- **Cursors are encrypted, not signed (decided)** — same code path as the
  spec 007 `requestState` blob, and repo-author-controlled cursors are never
  exposed, so no "don't put secrets in cursors" doc caveat is needed.
- **MRTR note** (spec draft, `basic/patterns/mrtr`): requirement 4 mandates
  *integrity* protection (HMAC or AEAD) for `requestState` — authenticated
  encryption satisfies it; no custom additional data is needed. Requirement
  5's SHOULDs (TTL, principal binding, originating-request binding) go
  *inside* the encrypted term — spec 007's payload concern; `max_age` covers
  the TTL part.
- **Default TTL: 20 minutes**, set at *encrypt* time via
  `Keyword.put_new(opts, :max_age, ...)`. `Plug.Crypto` embeds the mint-time
  `:max_age` inside the encrypted payload (`{data, signed_at, max_age}`) and
  `decrypt` enforces it when given no explicit `:max_age` — expiry is a
  tamper-proof mint-time policy, decrypt call sites cannot forget it.
- **Units gotcha**: `Plug.Crypto` / `Phoenix.Token` `:max_age` is in
  **seconds**. The old suite passed `to_timeout(hour: 2)` (milliseconds) to
  `Plug.Crypto.verify`, so its "2h" cursor expiry was really ~83 days. The
  rewrite must pass plain seconds.
- Key rotation: deferred. `secret_key_base` rotation is an app-level concern
  and can later be supported by accepting a list of key sources.

## Tests

- **`test/gen_mcp/token_test.exs` (passing 2026-06-13)** pins the whole
  `GenMCP.Token` API above: round-trips for every key-source shape (string,
  endpoint module, channel — including channel/raw interchangeability, i.e.
  cross-node verifiability), payload opacity (encrypted, not just signed),
  forged/tampered/wrong-key rejections, cursor↔method binding,
  cursor↔reqstate kind separation, host-app `Phoenix.Token` salt-namespace
  separation, nil → `:missing`, expiry. Test-local key sources: a fixed
  key-base string and a `FakeEndpoint` module exporting
  `config(:secret_key_base)` — no app config involved.
- Observable-level tests already written in `test/gen_mcp/suite_test.exs` (spec
  004 rewrite): cursors survive across fresh per-request states; a made-up
  cursor is rejected with `:invalid_cursor` (`-32602`). Under the decided design
  their `build_channel()` must set the channel `endpoint` — to be done when
  un-skipping them, not part of this task's test work.
