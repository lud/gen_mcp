# New 2026 protocol semantics

The behavioural changes that ride on top of the stateless core (specs 003-004).
Single-version now, so no version gating -- these are just the rules.

## Protocol-version header validation (`UnsupportedProtocolVersionError`, -32004)  — IN PROGRESS

`StreamableHTTP.validate_version_header/2` conflated two distinct failures in a
single catch-all `_ -> {:header_missing, ...}`:
- header **absent** (`[]`) -> correctly "missing" (-32001 HeaderMismatch);
- header **present but unsupported** (e.g. `["2025-06-18"]`) -> wrongly reported
  as *missing*, even when it agrees with the body `_meta`.

Per the draft Streamable HTTP spec, an unsupported version (unknown, or a known
version the server chose not to implement) MUST return **HTTP 400 +
`UnsupportedProtocolVersionError` (code `-32004`)** whose `data` lists the
supported versions. The schema (`$defs.UnsupportedProtocolVersionError`, `:nogen`
in codegen -- only an `error`/`allOf` envelope, no useful struct) requires
`data.requested` (string) and `data.supported` (string array).

**Fix (user):**
- `error.ex`: re-point `@mcp_unsupported_protocol_version` from `-32_000` to
  `-32_004`; add a caster:
  ```elixir
  defcasterror {:unsupported_protocol_version, requested},
      @mcp_unsupported_protocol_version, 400 do
    %{
      data: %{requested: requested, supported: GenMCP.supported_protocol_versions()},
      message: "Unsupported protocol version"
    }
  end
  ```
- `streamable_http.ex`: split the catch-all so order is
  `[@supported_vsn | _]` (then body-agreement check) -> `[vsn | _]`
  (`{:unsupported_protocol_version, vsn}`) -> `[]` (`{:header_missing, ...}`).

(Possible future cleanup, out of scope: a general "named errors" facility so
schema-defined error types like this don't have to be hand-rolled in `error.ex`.)

**Status:** test written (red) -- `streamable_http_test.exs` "rejects a request
whose MCP-Protocol-Version is a known-but-unsupported version". Code pending (user).

## Multi round-trip: `InputRequiredResult` (SEP-2322) — MOVED to spec 007

The full MRTR design (server reply tag, channel-assigns → encrypted
`requestState`, delta tracking, AEAD envelope, dispatch resume) lives in **spec
007** (draft). It is a whole subsystem, not a one-off semantic, so it was split
out of 005. 005 only needs to be aware that the `{:input_required, …}` reply

## `notifications/initialized` no-op (SEP-1442)  — DONE

A transitional client may still send `notifications/initialized`. The server
MUST accept-and-ignore it. Already implemented: the transport short-circuits it
before validation/dispatch (`suite.ex:158-159` generic no-op clause), and it is
covered by `streamable_http_test.exs:896` ("initialized notification is accepted
as a no-op with 202, without invoking the server"). No further work.

## Cache hints `ttlMs` / `cacheScope`  — IN PROGRESS

Confirmed present in V2607: the list/read results **require** (and `@enforce_keys`)
`cacheScope` + `ttlMs` (modeled on HTTP `Cache-Control`). Populate them on
list/read results where the Suite/repos can express caching; emit per provider.

**API decision (2026-06-16): flat builder opts.** Each list/read builder gains a
trailing optional opts keyword recognizing **flat** `cache_scope:` and `ttl_ms:`
(no nesting):
- Omitted -> `cacheScope: :private, ttlMs: 0` (today's no-cache, unchanged).
- Passing only one of the two fills the other with that no-cache default.
- `read_resource_result/1` is now **flat, arity-1**: a single keyword list.
  `split_mixed_props` peels off `:cache_scope`/`:ttl_ms`; a single content is
  built from `:uri`/`:mime_type`/`:blob`/`:text`, OR pre-made content
  structs/maps are passed as a list under `:contents` (in which case
  `:uri`/`:text`/`:blob` are ignored). E.g.
  `read_resource_result(uri: ..., text: ..., cache_scope: :public, ttl_ms: 60_000)`
  or `read_resource_result(contents: [...], cache_scope: :public)`.
- `discover_result` is a capability snapshot, **not** user content -> stays
  no-cache, no cache opts.

**Status:** builders implemented (constants renamed `@default_cache_scope` /
`@default_ttl_ms`); all builder + flat-API tests green
(`mcp/v2607_test.exs`).

### `read_resource_result` `:contents` edge cases

Tested (green): `:contents` precedence over `:uri`/`:text`/`:blob`; empty
`:contents` -> `contents: []` (schema has no `minItems`); per-content `:_meta`
preserved when built via `resource_contents/1` inside `:contents`.

Guard (done): a non-list `:contents` (single struct, unwrapped) now raises a
helpful `ArgumentError` instead of a raw `CaseClauseError`. The other footgun --
a keyword list (`contents: [uri: ..., text: ...]`) silently passed through as
`contents: [{:uri, _}, ...]` -- is **left to the typespec + docs** rather than a
runtime guard (decision 2026-06-16): the `@spec` flags the misuse statically and
proper docs will make correct usage clear, so no runtime check.

`:_meta` semantics (resolved 2026-06-17): a top-level `:_meta` is split off as a
wrapper opt (`Keyword.split(opts, [:cache_scope, :ttl_ms, :_meta])`) and set on
the **`ReadResourceResult` object**, not on the content -- by design, matching
the docstring. Per-content `_meta` is attached via `resource_contents/1` inside
`:contents`. Both paths covered: `mcp/v2607_test.exs` "top-level :_meta feeds the
ReadResourceResult, not the content" / "... even when :contents is given" /
":_meta is preserved per content ...".

Builders to update (arity bump): `list_tools_result/1 -> /2`,
`list_resources_result/2 -> /3`, `list_resource_templates_result/1 -> /2`,
`read_resource_result/1 -> /2`, `list_prompts_result/2 -> /3`.

**Status:** builder-level tests written (`mcp/v2607_test.exs`, describe
"cache hints (spec 005)") -- 5 default-behaviour cases green, 7 explicit-hint
cases red until the builders accept the opt. Code change pending (user).

### Suite-level wiring (decided 2026-06-17)  — DONE

**Status (2026-06-17):** implemented and green (full suite 290 passing). Each
provider behaviour (`Tool` / `ResourceRepo` / `PromptRepo`) has an optional
`cache_control/1` + a guarded wrapper (`function_exported?` -> `MCP.default_cache_control/0`).
`resources/list` / `prompts/list` use the producing repo's hint per page;
`tools/list` / `templates/list` combine via `merge_cache`/`cache_opts`. Mock
skips applied (`ToolMock`, `ResourceRepoMockTpl`, `PromptRepoMock`).

**Categories.** Only results the *Suite* builds need a hint source:
- **Passthrough (no work):** `resources/read` -- the `ResourceRepo.read/3`
  callback builds the `ReadResourceResult` itself and sets cache opts directly.
- **Not cacheable at all:** `tools/call` / `prompts/get` -- `CallToolResult` /
  `GetPromptResult` have no cache fields. Ignore.
- **Suite builds (needs a source):** the four list ops -- `tools/list`
  (suite.ex:254), `resources/list` (312), `resources/templates/list` (354),
  `prompts/list` (393). (`server/discover` also builds one but stays no-cache.)

**Mechanism.** A single optional callback on the contributing provider
behaviours:
```elixir
@callback cache_control(arg) :: {:public | :private, non_neg_integer()}
@optional_callbacks cache_control: 1   # add on Tool, ResourceRepo, PromptRepo
```
Channel-less, matching `Tool.info/2` / `input_schema/1` style (tool descriptions
are built without a channel).

**Two shapes (decided 2026-06-17), because pagination changes the picture:**

- **Paginated, one repo per page -> NO combination.** `resources/list` and
  `prompts/list` go through `paginate_repos`, which returns on the *first*
  non-empty repo: a page's items always come from a **single** repository
  (`suite.ex:456-474`). So a page just carries **that repo's** `cache_control`.
  Different pages may carry different hints -- correct, since each page is its
  own `CacheableResult` (per-response cache, HTTP `Cache-Control` analogy); the
  spec does not require a paginated listing to share one policy across pages.
  Implementation: `paginate_repos`/the list closure must surface *which* repo
  produced the page (the `repo` is already in scope) so the caller reads its
  hint. Empty page (no producing repo) -> `{:private, 0}`.

- **Aggregated into one result -> COMBINE.** `tools/list` (all tools) and
  `resources/templates/list` (all template repos) build a single result, so the
  Suite combines across contributors:
  - a contributor that does not export `cache_control` contributes `{:private, 0}`;
  - `scope = :public` iff EVERY contributor is `:public`, else `:private`;
  - `ttl = max(0, min(ttls))`;
  - empty -> `{:private, 0}`.
  (A catalog is only as cacheable as its least-cacheable part.)

Most real deployments have one extension + one repo, so combination rarely bites.
**Add a docs TODO** for the future guide when implementing.

**Tests (red until wired):** `suite_test.exs` describe "cache hints wiring (spec
005)" -- 8 tests. Paginated/per-page: resources/list page hint, resources/list
per-page variation (different hint each page), resources/list default no-cache,
prompts/list page hint. Aggregated/combine: templates/list public-iff-all+min,
templates/list private-if-any, tools/list public-iff-all+min, tools/list
non-implementer poisons. New mocks (`test/support/mocks.ex`):
`ResourceRepoCacheMock`, `ResourceRepoTplCacheMock`, `PromptRepoCacheMock`.

**Compile-coupled (must land WITH the callback declaration):** once
`cache_control` is optional, every non-skipping mock auto-implements it and would
break existing Suite list tests. Add `cache_control: 1` to the skip lists of:
`ToolMock`, `ResourceRepoMockTpl`, `PromptRepoMock`. (`ResourceRepoMock` already
skips all optionals; the `*CacheMock`s and `ToolFullMock` intentionally keep it.)

## Client disconnect → `handle_close` (DONE 2026-06-17)

**Decision:** add an optional `handle_close` cleanup hook, minimal form.
Implemented and green across all layers (full suite 294 passing): server-level
`GenMCP.handle_close/2` + worker `:CHAN_DOWN` wiring, Suite/Tool mirror
(`continue/4` -> `handle_message/4`, new `handle_close`), and a real-HTTP
disconnect test on `/mcp/mock` (`ServerMock` is now the full default mock;
`ServerMockNoClose` covers the "not implemented -> stop immediately" path).

**Server-initiated close — the `:closed` ACK design (decided 2026-06-17).** A
handler may end the stream itself via `Channel.close/1`. This does **NOT** ride
`:CHAN_DOWN`: on a keep-alive connection the relay (Bandit conn process) survives
a `finalize`, so the worker's monitor never fires for a server-side close (and
the worker would leak waiting for it). Why a monitor can't cover this: the worker
monitors `channel.client` (the conn process); `:CHAN_DOWN` means *that process
died* = socket loss = client disconnect only.

Instead, the close is acknowledged worker-ward:
- `Channel.close/1` -> `{:"$gen_mcp", :close}` to the conn process; the handler
  returns `{:stream, _}` so the worker stays alive for the ack.
- new `stream_loop` clause: `{:"$gen_mcp", :close} -> send(worker, {:"$gen_mcp",
  :closed}); finalize(conn)`. The worker pid is stashed in `conn.private` at
  `dispatch_req` (the conn already monitors it). The ack is sent **only** from
  this clause — not unconditionally from `finalize` — so normal completions never
  ping the worker.
- new worker `handle_info({:"$gen_mcp", :closed}, _)` -> `set_closed` the channel,
  run `handle_close` iff exported, stop `{:shutdown, :closed}`.

`:CHAN_DOWN` handling **stays** (client disconnect still happens); the two paths
both call `handle_close` and don't double-fire (the worker stops on whichever
arrives first). A deliberate `{:stop, _}`/`{:result, _}` does **NOT** fire
`handle_close` — that's an orderly termination the handler already controls.

Rejected: calling `handle_close` from a `{:stop, _}` return (terminate-style) —
`handle_close` is for closes the handler didn't drive to a result, not every
exit. Also rejected: sending the ack unconditionally from `finalize` (fires for
nothing on the ~90% normal-completion case).

**Status / tests:**
- Worker side (user): add the `{:"$gen_mcp", :closed}` `handle_info`. Driven by
  `server_test.exs` "the worker runs handle_close on a :closed ack" (RED — the
  worker has no `:closed` clause yet, so it falls through to `handle_message`).
- Transport side (user): the `dispatch_req` worker-pid stash + the `:close`
  `stream_loop` clause. Driven end-to-end by `streamable_http_test.exs` "a
  server-initiated Channel.close ..." (RED — `:close` currently hits
  `unexpected_message`, and `handle_close` never fires).

**Detection-latency caveat (flagged, not blocking):** the StreamableHTTP relay
detects a dropped socket only on the **next chunk write** — a `:keepalive` every
`@stream_keepalive_timeout` (25s) or an emitted event. So for an *idle* streaming
request (a long-lived listener that rarely writes), `:CHAN_DOWN` — and therefore
`handle_close` — can lag up to ~25s after the client actually disconnects. Fine
for request/response streams; relevant for **spec 008** (`subscriptions/listen`
long-lived streams holding resources). If prompter cleanup is needed, the
transport would need proactive socket-close detection (e.g. handling the
Bandit/`tcp_closed` message in `stream_loop` rather than ignoring it). The
disconnect test sidesteps this by having the server tick ~every 50ms.

Server level (`GenMCP` behaviour) — DONE (scaffolding + tests):
- `@callback handle_close(Channel.t(), state) :: term`, `@optional_callbacks
  handle_close: 2`. Same args as `handle_message/3` **minus the message**, so an
  impl can delegate: `def handle_close(c, s), do: handle_message(:client_closed, c, s)`.
- On `:CHAN_DOWN` the worker (user-implemented) marks the channel `:closed`,
  calls `handle_close` **iff exported**, ignores the return, then stops
  gracefully. **Not implemented -> stop immediately** (prior behaviour).
- Tests: `server_test.exs` "stops immediately ... handle_close not implemented"
  (green) and "invokes handle_close with a closed channel before stopping"
  (red until the worker calls it). `gen_mcp_test.exs` callback list updated.
  Mocks: `ServerMock` skips `handle_close: 2`; new `ServerFullMock` implements it.
- `TODO(spec 005)` left in the `handle_close` `@doc` to document it in the guides.

Suite/Tool level — DONE (decided: full mirror, 2026-06-17):
- Tool callback `continue/4` **renamed** to `handle_message/4` repo-wide (behaviour
  `@callback`, the `Tool.handle_message/4` wrapper, `Suite` call site
  `handle_tool_message`, docs, mocks, tests) so the Suite tool callbacks mirror
  the server's `handle_message`/`handle_close`.
- New optional Tool `@callback handle_close(channel, state, arg)` + a guarded
  `Tool.handle_close/3` wrapper (no-op when unexported).
- `Suite.handle_close/2` (the `GenMCP` callback impl) forwards to the active
  streaming tool's `handle_close`, mirroring `handle_message/3`. Tested green
  (`suite_test.exs` "handle_close forwards ..." / "no-op when no streaming tool").
- Mocks: `ToolMock` skips `handle_close: 3` (keeps `handle_message: 4`);
  `ToolFullMock` implements both.

**Only remaining piece (user):** the worker change in `GenMCP.Server` —
`handle_info({:CHAN_DOWN, ...})` should mark the channel `:closed`, call
`handle_close/2` iff exported, then stop. Its test is red:
`server_test.exs` "invokes handle_close with a closed channel before stopping".

## Client disconnect → server implementation (original notes)

On the Streamable HTTP transport a client disconnect **is** the cancellation
signal (spec 004): the relay's `spawn_monitor` fires and `GenMCP.Server` receives
a `:CHAN_DOWN` message. Today the worker intercepts it and stops **without**
notifying the server implementation:

```elixir
# lib/gen_mcp/server.ex:151
def handle_info({:CHAN_DOWN, mref, :process, _pid, _reason}, %{mref: mref} = state) do
  {:stop, {:shutdown, :client_disconnected}, state}
end
```

Every other message is routed to `server_mod.handle_message/3`
(`server.ex:155-163`); the disconnect alone is swallowed by the worker. The
Suite's `handle_message/3` forwards messages to a streaming tool's `continue/4`,
so a tool currently never hears about the disconnect — the worker just exits and
anything it linked / monitored / registered is torn down with it.

**Question:** should `GenMCP.Server` deliver the disconnect (the raw `:CHAN_DOWN`
or a normalized signal) to `handle_message/3` before/instead of stopping, so the
implementation can run explicit cleanup?

- **Today's behaviour is fine** when tools rely on automatically-cleaned
  processes (links, monitors, `Task`, registration): the worker exit cascades and
  cleans everything up.
- **It is insufficient** when a tool holds resources that are *not* auto-cleaned
  (external handles, manual registry entries, side effects to undo): such a tool
  has no hook to release them on disconnect.

Open sub-questions (not deciding now):

- **Shape:** raw `:CHAN_DOWN` vs a normalized `{:disconnected, reason}` /
  `:cancelled` message to `handle_message/3`, vs a dedicated `terminate`-style
  callback distinct from message handling.
- **Return semantics:** after a disconnect there is no stream left to write to, so
  any `{:result, _}` / `{:stream, _}` return is moot — delivery would be purely
  for side-effecting cleanup, and the worker still stops afterwards.
- **Scope:** only a streaming request (one that returned `{:stream, _}` and has a
  live `continue/4`) can meaningfully observe this; a terminated request has no
  worker left to notify.

**Status: undecided** — captured so the stateless core does not silently
foreclose user cleanup on client disconnect.

Possible solutions:
- Do nothing
- Add an optional handle_close callback. In Suite, define a terminate or disconnected callback for tools. (Also maybe we should rename continue to handle_message, and use the same handle_close name here). The return values are ignored (`term`), the GenServer exits gracefully in all cases.

**Recommendation (2026-06-16, pending user confirmation):** add the optional
`handle_close` callback (minimal form), because client disconnect is the *normal*
cancellation path on stateless Streamable HTTP, and long-running streaming tools
are exactly the ones likely to hold resources not tied to the worker's process
lifecycle (remote handles, reserved slots, rows to mark abandoned). Constraints:
- **Optional**, default no-op -- backward compatible.
- **Scoped to streaming requests only** (a request that already returned has no
  live worker to notify).
- **Return value ignored** (`term`); worker exits gracefully regardless -- the
  hook is purely for side-effecting cleanup, never for writing to the dead stream.
- Treat the `continue/4` -> `handle_message/4` **rename as separate/optional**,
  not a blocker for the hook (wider churn across the Suite streaming contract).
Sequencing: do this **last** in 005, after the decision-free items land.
