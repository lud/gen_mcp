# New 2026 protocol semantics

The behavioural changes that ride on top of the stateless core (specs 003-004).
Single-version now, so no version gating -- these are just the rules.

## Error code: missing resource `-32002` -> `-32602`

`lib/gen_mcp/error.ex:39` -- `@mcp_resource_not_found` is `-32_002`. Change to
`@rpc_invalid_params` (`-32_602`). No version branch needed (single version).
Note `@mcp_prompt_not_found` (line 40) is already `-32602`. Remove the now-dead
`@mcp_unsupported_protocol_version` (line 43) handshake error and the
session-related casts (`:missing_session_id`, `:unexpected_session_id`,
`:session_not_found`, `:mcp_session_restore_failure`, lines 45-55, 174-185) once
specs 003-004 delete their callers.

## Multi round-trip: `InputRequiredResult` (SEP-2322) — MOVED to spec 007

The full MRTR design (server reply tag, channel-assigns → encrypted
`requestState`, delta tracking, AEAD envelope, dispatch resume) lives in **spec
007** (draft). It is a whole subsystem, not a one-off semantic, so it was split
out of 005. 005 only needs to be aware that the `{:input_required, …}` reply
returns a plain JSON result (never a stream); everything else is 007's concern.

## `notifications/initialized` no-op (SEP-1442)

A transitional client may still send `notifications/initialized`. The server
MUST accept-and-ignore it. `Suite.handle_notification/2` already has a clause
(`suite.ex:339`) -- keep it as a no-op; ensure the stateless path routes
notifications there rather than erroring on unknown method.

## Tasks via the external extension (SEP-2663)

Tasks are **not** in core -- they live in `io.modelcontextprotocol/tasks` (own
`ext-*` repo + schema). `tasks/list` does not exist.

- Vendor the tasks extension schema as its own SHA-pinned dep; generate its
  types into a dedicated namespace (e.g. `GenMCP.MCP.Ext.Tasks`).
- New `lib/gen_mcp/suite/tasks.ex` implementing `tasks/get` / `tasks/update` /
  `tasks/cancel`; `tools/call` may answer with a task handle.
- Wire into `Suite.do_handle_request/3` **behind capability negotiation** -- only
  when the client advertises the tasks extension. Never always-on.
- Reuse the existing async-tool `tracker` record (`suite.ex:102`) as the backing
  for task handles.
- **Scope check:** this is the largest sub-piece. If bandwidth is tight, ship
  the stateless core (002-004 + errors/InputRequired/no-op) first and treat
  Tasks as a fast-follow -- it is capability-gated, so omitting it is safe.

## Cache hints `ttlMs` / `cacheScope`

Confirmed present in V2607: the list/read results **require** (and `@enforce_keys`)
`cacheScope` + `ttlMs` (modeled on HTTP `Cache-Control`). Populate them on
list/read results where the Suite/repos can express caching; emit per provider.

## Tests

- Resource-not-found returns `-32602`.
- Stray `notifications/initialized` is accepted with no error.
  (`InputRequiredResult` round-trip tests live in spec 007.)
- Tasks lifecycle (if implemented): call -> handle -> get -> update -> cancel;
  `tasks/list` rejected as unknown method.

## Acceptance criteria

- RC semantics observable end-to-end on the stateless core.
- Tasks either implemented behind capability negotiation or explicitly deferred
  as a documented fast-follow.
