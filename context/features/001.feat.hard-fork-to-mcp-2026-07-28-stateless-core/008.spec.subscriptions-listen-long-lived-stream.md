# `subscriptions/listen`: the long-lived SSE stream

> Deferred out of spec 003 so the transport rework stayed focused on stateless
> request/response. Design resolved 2026-06-18 (see "Design decisions" below).
> This is the **only surviving server-push path** after the standalone GET SSE
> stream was removed in 2026-07-28.

## What it is

Spec sources (verified 2026-06-19):
- Normative subscription rules:
  <https://modelcontextprotocol.io/specification/draft/basic/patterns/subscriptions>
- HTTP stream mechanics:
  <https://modelcontextprotocol.io/specification/draft/basic/transports/streamable-http>

The 2026-07-28 revision **removed the standalone GET SSE stream**. Long-lived
server→client notifications now require an explicit opt-in request:

> "Long-lived notification streams are obtained by sending a `subscriptions/listen`
> request. The server's response is itself an SSE stream that stays open and
> delivers the change notifications the client opted in to (such as
> `notifications/tools/list_changed` or `notifications/resources/updated`)."

Contrast with **request-scoped** notifications (`notifications/progress`,
`notifications/message`): those flow **only** on the response stream of the
request they relate to and are handled in spec 003 — they are **not** delivered on
the listen stream.

## Why it is its own spec

- It is a **genuinely long-lived connection**, unlike the per-request SSE in 003
  (which lives for the duration of one request). Different lifecycle, keepalive,
  and teardown concerns.
- Without it, list-change / resource-update notifications simply do not reach
  clients. That is an acceptable temporary gap while 003/004/005 land — the
  stateless core works without it — but it must be tracked.

## Key finding: the plumbing already exists

The stateless transport/behaviour layers need **nothing new**. A
`subscriptions/listen` request is, mechanically, just a request whose handler
opens a stream and keeps it open. Every callback the deleted `SessionController`
used for listening maps directly onto a callback that already survived the fork:

| Old (session `SessionController`, on `main`)                                              | New (stateless, current)                                             | Role                           |
| ----------------------------------------------------------------------------------------- | -------------------------------------------------------------------- | ------------------------------ |
| `listener_change/2` (stream opens) — *"the right place to setup subscriptions to pubsub"* | `handle_request(SubscriptionsListenRequest)` → `{:stream, state}`    | subscribe + commit SSE         |
| `handle_info/3` — receive pubsub msg, push to channel                                     | `handle_message/3` — doc already names *"subscribing to a producer"* | event → `Channel` notification |
| `listener_change/2` (stream closes)                                                       | `handle_close/2`                                                     | teardown                       |

The 003 `stream_loop` / 25s keepalive and `GenMCP.Mux.Channel.send_*` plumbing
already deliver everything. The *request* vocabulary is already generated (spec
002): `GenMCP.MCP.V2607.SubscriptionsListenRequest` /
`SubscriptionsListenRequestParams`, `SubscriptionFilter`,
`SubscriptionsAcknowledgedNotification`. **No transport change.**

> **Update 2026-06-19 (codegen needed after all).** The *change-notification*
> structs the stream actually delivers — `ToolListChangedNotification`,
> `ResourceListChangedNotification`, `PromptListChangedNotification`,
> `ResourceUpdatedNotification` (+ `…Params`) — were `:nogen` in
> `tools/gen-rpc-schemas.exs` and so did not exist. They are now enabled (`[]`)
> and generated via `just gen-entities` (all refs resolve to already-generated
> defs — `NotificationParams`, `NotificationMetaObject`, `RequestId`). Also note
> the schema **now** declares `io.modelcontextprotocol/subscriptionId` as a
> typed `RequestId` field on `NotificationMetaObject` (it did not when this spec
> was first drafted), so the stamped key has a typed home on the generated
> structs.

## Design decisions

### gen_mcp owns no pub/sub (agnostic plumbing)

The producer of `tools/list_changed` / `resources/updated` is the
**application**, not the framework. gen_mcp ships only the stream lifecycle; the
app broadcasts changes on whatever pub/sub it already uses (`Phoenix.PubSub`,
`:pg`, `Registry`, a `GenServer`, …) and translates received events into
notifications. No dependency is added. Convenience helpers may grow over time,
but are out of scope here.

### Custom `GenMCP` servers: just `handle_request`

A server implementing the `GenMCP` behaviour directly receives
`subscriptions/listen` in `handle_request/3` like any other request and may do
whatever it wants — typically subscribe to its producer, send the
`acknowledged` notification, and return `{:stream, state}`. This is the intended
path for **small servers** (a few tools/resources, everything in one module).
**No new API is required for them.** Example:

```elixir
def handle_request(%MCP.SubscriptionsListenRequest{params: %{notifications: filter}}, channel, state) do
  :ok = MyApp.PubSub.subscribe("tools")
  channel = Channel.send_notification(channel, ack(filter))
  {:stream, state}
end

def handle_message({:tools_changed}, channel, state) do
  Channel.send_notification(channel, %MCP.ToolListChangedNotification{})
  {:stream, state}
end
```

### `GenMCP.Suite`: a dedicated `SubscriptionHandler` behaviour

This spec is mostly for `Suite`. As a `GenMCP` behaviour implementation, `Suite`
receives `subscriptions/listen` in its own `handle_request/3` (where it already
dispatches per request type) and routes it to a new behaviour,
**`GenMCP.Suite.SubscriptionHandler`**.

Naming: `SubscriptionHandler` (not `Listener`) — it does not necessarily listen
to anything; it *accepts a subscription* and *may send notifications*. Compound,
precise.

It is a **dedicated** behaviour that **structurally mirrors `Suite.Tool`** but is
not a reuse of it. Tool already carries the same streaming triad (`call` entry /
`handle_message/4` / `handle_close/3`), so a Suite author who has written a Tool
will find `SubscriptionHandler` shaped identically — only the entry name and the
return vocabulary differ. A `use Suite.SubscriptionHandler` that stubbed the Tool
callbacks was considered and rejected: notification handling will likely evolve
separately from tools, and conflating the two invites mistakes.

### Callback API rationale

The draft first tried a GenServer-flavoured `init`/`{:ok}` entry. Dropped: it
created a **hybrid** (`init`→`{:ok}` from GenServer, `handle_message`→`{:stream}`
from gen_mcp) where every callback half-reminds you of a different model, so
muscle memory fights at each one. Committing fully to GenServer would force
`handle_info/3` and `terminate/3` — arity-3 shapes that *look* like GenServer and
then break its `/2` muscle memory (channel must be threaded, gen_mcp house
style).

So we commit to the **`Suite.Tool` model** instead: it is in the same library,
the developer just learned it, and threading `channel` per callback is already
its convention. The entry is named `subscribe` (accurate — it accepts a
subscription; also sidesteps any confusion with `GenMCP.init/1`, though no module
implements both behaviours). The return vocabulary is **narrowed** to
`{:stream, state} | {:stop, reason}` — a subscription never produces a
`{:result, …}`, so that arm (which Tool needs) is dropped.

```elixir
defmodule GenMCP.Suite.SubscriptionHandler do
  @doc """
  Accept (or reject) a subscription. Runs in the stream-handling process; set up
  the app subscription here (pub/sub, GenStage, …).

  * `{:stream, state}` — accept as requested: opens the long-lived SSE stream and
    honors the **full requested filter**. Suite then emits the
    `notifications/subscriptions/acknowledged` message before entering the stream
    loop.
  * `{:stream, honored, state}` — accept but **downgrade**: honor only `honored`
    (auth/capability). Suite intersects it with the requested filter (it can only
    narrow, never widen) and reports the effective set in `acknowledged`.
  * `{:stop, reason}` — reject: no stream is opened, a normal error response is
    returned instead.
  """
  @callback subscribe(filter :: SubscriptionFilter.t(), Channel.t(), arg) ::
              {:stream, state}
              | {:stream, honored :: SubscriptionFilter.t(), state}
              | {:stop, reason :: term}

  @doc "An app message (pub/sub delivery, …) → notifications on the channel."
  @callback handle_message(msg :: term, Channel.t(), state, arg) ::
              {:stream, state} | {:stop, reason :: term}

  @doc "Teardown when the client closes the stream (close = cancel)."
  @callback handle_close(Channel.t(), state, arg) :: term
end
```

Structurally `subscribe`/`handle_message`/`handle_close` ≡ Tool's
`call`/`handle_message`/`handle_close`, with the narrower
`{:stream, state} | {:stop, reason}` return.

### The acknowledgment is mandatory — Suite owns it

The spec (`patterns/subscriptions`) is normative: the server **MUST** send
`notifications/subscriptions/acknowledged` as the **first message** on the
stream, and the ack's `notifications` field reports the subset the server agreed
to honor.

Because it is a MUST and the agnostic handler could forget, **Suite always sends
the ack itself**, unconditionally — for both `{:stream, state}` and
`{:stream, honored, state}`. The handler never sends it. The return shape only
decides *what* the ack reports:

- `{:stream, state}` (no payload) ⇒ ack reports the **full requested filter**
  ("accept exactly what the client asked for" — the common case).
- `{:stream, honored, state}` ⇒ ack reports `honored` (deliberate downgrade for
  auth/capability).

**Filter enforcement is the handler's responsibility for now.** The spec says the
server MUST NOT push a type the client did not request, but Suite does **not**
intersect/validate or drop non-honored notifications at send time — we trust the
handler (the filter is a handful of booleans + a URI list; enforcement code is
fiddly and may get in the way). This can be added later without an API change.

### Every stream message carries `subscriptionId`

The spec requires every message on the stream — the ack and each change
notification — to carry `io.modelcontextprotocol/subscriptionId` in
`params._meta`, where the value is the **JSON-RPC id of the
`subscriptions/listen` request** (so a client can demultiplex concurrent
subscriptions). On HTTP this is redundant (each request has its own response
stream) but still a MUST; it earns its keep on stdio, where all messages share
one channel.

The schema in `deps/modelcontextprotocol/schema/draft/schema.json` does **not**
declare this `_meta` key, so it is **not** a typed field on the generated
entities — it must be injected dynamically into `_meta`.

#### `Channel` gains a `request_id`

Since the value is just the JSON-RPC request id, add a `request_id` field to
`GenMCP.Mux.Channel`, set by the transport from the request id for **every**
request (not only `subscriptions/listen`). Benefits beyond subscriptions: repos
(`ResourceRepo` / `PromptRepo`) and tools receive only the channel, not the
original request, so `channel.request_id` gives them the id for free.

- `from_request/3` reads the JSON-RPC `id`.
- Notification-built channels and `for_pid/2` leave it `nil` — correct, since a
  JSON-RPC notification has no id.
- Adding it touches `@enforce_keys` and all constructors.

#### `Channel.send_notification/2` stamps it defensively

A new `Channel.send_notification(channel, payload)` is the generic sender for
arbitrary notifications (the subscription change notifications go through it). It
injects `channel.request_id` as `io.modelcontextprotocol/subscriptionId` into
`payload`'s `params._meta`. Because callers may pass **raw maps** (users are not
required to use the schema-derived structs), it must be defensive:

- Accept payloads as **structs** (atom keys), **atom-keyed maps**, or
  **string-keyed maps**.
- At each level (`params`, `_meta`, the key itself) **detect the existing key
  form** (atom `:_meta` vs string `"_meta"`, etc.) and merge in that form;
  create the container if absent.
- **Never overwrite** an already-present `subscriptionId` (or the `_meta` map) —
  even if it looks wrong; only fill it in when missing.
- If `channel.request_id` is `nil`, send as-is (no stamping).

**Immediate behavior:** stamp **unconditionally** when `request_id` is present
(not gated to listen streams). `request_id` exists on every channel, but
`send_notification/2` is documented as the long-lived/subscription-notification
path; progress/log use their own functions. On HTTP-focused gen_mcp this is the
only real caller, so the simpler always-stamp wins.

**Follow-up now enabled by codegen:** stamp only when the payload is actually a
subscription-stream notification — detect it from the struct module or the
`method` field (`:method` / `"method"`) and stamp only for the watched set
(`notifications/tools/list_changed`, `notifications/resources/list_changed`,
`notifications/prompts/list_changed`, `notifications/resources/updated`,
`notifications/subscriptions/acknowledged`). That watched set is now generated
from the upstream schema into `GenMCP.MCP.V2607.Info.subscription_notification_methods/0`,
so the future `Channel.send_notification/2` refinement can consume it without a
hand-maintained list.

Suite uses `send_notification/2` for the ack and forwards handler notifications
through it (or the handler calls it directly), so `subscriptionId` is stamped in
one place for both the Suite and custom-server paths.

### Suite wiring

On `subscriptions/listen`, `Suite` calls the configured handler's `subscribe/3`,
sends the `acknowledged` notification (stamped with `subscriptionId`) reporting
the honored filter, returns `{:stream, ...}` to the runner, forwards subsequent
process messages to `handle_message/4`, and calls `handle_close/3` on disconnect.

### Capability advertisement (`server/discover`)

> **Resolved 2026-06-20.** The 2026-07-28 `ServerCapabilities` (verified in
> `deps/modelcontextprotocol/schema/draft/schema.json`) has **no dedicated
> top-level `subscriptions` capability** — the standalone GET-SSE removal took it
> with it. Subscription support is advertised purely through **sub-flags on the
> existing catalog blocks**:
>
> | Sub-flag             | Notification advertised                  |
> | -------------------- | ---------------------------------------- |
> | `tools.listChanged`     | `notifications/tools/list_changed`     |
> | `prompts.listChanged`   | `notifications/prompts/list_changed`   |
> | `resources.listChanged` | `notifications/resources/list_changed` |
> | `resources.subscribe`   | `notifications/resources/updated`      |

gen_mcp is pub/sub-agnostic and the handler decides the honored subset *per
subscription* at `subscribe/3` time, so `Suite` cannot infer statically which
types a handler emits. The handler therefore **declares** them through a new
**optional** callback on `SubscriptionHandler`:

```elixir
@callback subscription_capabilities(Channel.t(), arg) :: %{optional(flag) => boolean}
# flag ::
#   :tools_list_changed      -> tools.listChanged
#   :prompts_list_changed    -> prompts.listChanged
#   :resources_list_changed  -> resources.listChanged
#   :resources_updated       -> resources.subscribe
```

- **Argument order follows the Suite convention** (subject first, `arg` last,
  `channel` second-to-last). `subscription_capabilities` has **no subject** — it
  is conceptually a near-constant function — so the convention leaves just
  `(channel, arg)`. This is *not* a "channel-first" rule. `arg` is the configured
  handler options (so generic handlers can vary by config); `channel` carries the
  request's read-only `meta` (client info / capabilities / auth context + copied
  assigns) so advertisement MAY vary by authentication — e.g. advertise
  `resources.subscribe` only to an authorized client. (Settled 2026-06-20; the
  first sketch was `arg`-only with no channel.)
- **Snake-case keys**, each mirroring its notification method
  (`:resources_updated` ⇒ `notifications/resources/updated` ⇒ the `subscribe`
  sub-flag — the schema names that flag `subscribe`, but it gates the
  `resources/updated` notification). Absent or `false` keys advertise nothing.
- **Optional; defaults to `%{}`.** A handler that does not implement it (or that
  returns `%{}`) advertises no subscription capability — current behavior is
  unchanged. The `-32601` backstop still covers a misbehaving client that sends
  `subscriptions/listen` to a server advertising nothing.

**Merge rule (`Suite.capabilities`).** A catalog block is advertised iff the
catalog has items of that kind **or** a sub-flag is declared `true` for it; the
block's sub-flags are the union of the declared-`true` flags. So a handler that
declares `tools_list_changed: true` on a server with no tools still advertises
`tools: %{listChanged: true}` (a list that may go from empty to populated is a
legitimate `list_changed` source), and a server with tools but no declared flags
keeps advertising the bare `tools: %{}` it does today.

`logging` is independent of all this and stays always-on.

## Resolved open questions

1. **Registry / "no GenServer" goal.** No central registry is introduced. The
   listen worker *is* the already-existing per-request `GenMCP.Server` process
   (a `DynamicSupervisor` child). It subscribes *itself* to the app's topic;
   with a subscriber-driven pub/sub the broadcaster holds **no list of
   streams**. State is **connection-scoped** (dies with the stream), never
   session-scoped. Boundary stated: connection-scoped ✅ / session-scoped ❌.
2. **Notification source in `Suite`.** The application, via the
   `SubscriptionHandler` behaviour. gen_mcp owns no pub/sub (see above).
3. **Keepalive / idle / max-duration.** Reuse 003's 25s keepalive. No idle
   timeout (the stream is deliberately long-lived). An optional configurable
   max-duration MAY be added; default is unbounded (client reconnects).
4. **Cancellation.** Close = cancel, via `handle_close`. Consistent with 003's
   per-request "close = cancel". Server-side teardown (e.g. shutdown) **MUST**
   close the SSE stream (the spec's HTTP teardown signal).
5. **Auth / scoping.** The `Channel` already carries client info / capabilities
   / auth context from `_meta`. `subscribe/3` authorizes and MAY downgrade the
   filter (`{:stream, honored, state}`) or reject (`{:stop, reason}`); the
   effective honored set is reported in the `acknowledged` notification.

## The `SubscriptionFilter` contract

`SubscriptionsListenRequestParams.notifications` is a `SubscriptionFilter`
(`GenMCP.MCP.V2607.SubscriptionFilter`):

- `toolsListChanged` (boolean) → `notifications/tools/list_changed`
- `resourcesListChanged` (boolean) → `notifications/resources/list_changed`
- `promptsListChanged` (boolean) → `notifications/prompts/list_changed`
- `resourceSubscriptions` (array of URI strings) →
  `notifications/resources/updated` for those URIs (replaces the former
  `resources/subscribe` RPC)

All fields are optional; omitting one means "not subscribed to that type". Each
type is **opt-in**: the server **MUST NOT** send a type the client did not
request.

The **first** message on the stream **MUST** be
`notifications/subscriptions/acknowledged`, whose `params.notifications` is the
**honored** filter (the subset the server agreed to; unsupported types omitted —
allows downgrade). Every message on the stream (the ack and each change
notification) **MUST** carry `io.modelcontextprotocol/subscriptionId` in `_meta`,
equal to the JSON-RPC id of the `subscriptions/listen` request — so a client can
demultiplex multiple concurrent subscriptions. On HTTP each `subscriptions/listen`
is its own POST with its own response stream, so concurrent subscriptions are
naturally separate streams (and separate per-request workers).

## Resolved against the updated schema (2026-06-20)

- **Do request-scoped notifications also need `subscriptionId`? — No.** The schema
  now declares the key on `NotificationMetaObject`
  (`deps/modelcontextprotocol/schema/draft/schema.json`) with a normative
  description: the server MUST include it on every notification delivered via a
  `subscriptions/listen` stream, and it is **absent** on notifications not
  delivered via a subscription stream — the description names *progress
  notifications for an in-flight request* as the example of where it must NOT
  appear. So `notifications/progress` and `notifications/message` (the dedicated
  `Channel.send_progress/4` / `send_log/4` paths) need **no stamping**, confirming
  the earlier "leaning: no". This **unblocks** the gate item.
- **Consequence for `send_notification/2`:** because the key MUST be absent on
  non-subscription notifications, gating the stamp to the watched subscription
  method set is no longer merely an optimization — it is the spec-correct
  behavior. Promoted from optional to the remaining implementation step (see the
  checklist).

## Acceptance criteria

- A client can `subscriptions/listen` with a filter and receive the opted-in
  change notifications on a long-lived SSE stream, **always** preceded by the
  `acknowledged` notification (first message) reporting the honored filter.
- The ack and every change notification carry
  `io.modelcontextprotocol/subscriptionId` (= the listen request's id) in
  `_meta`.
- Closing the stream cleanly tears down the subscription (`handle_close`) with no
  leaked process.
- No per-session registry is reintroduced (connection-scoped state only).
- A custom `GenMCP` server can implement `subscriptions/listen` entirely in
  `handle_request/3` with no `Suite`-specific API.

## Testing notes (tests authored by Claude, implementation by user)

- `acknowledged` is **always** the first stream message, for both `{:stream,
  state}` and `{:stream, honored, state}`.
- `{:stream, state}` (no payload) ⇒ ack reports the full requested filter;
  `{:stream, honored, state}` ⇒ ack reports `honored`.
- The ack and every notification carry `io.modelcontextprotocol/subscriptionId`
  in `params._meta` equal to the listen request's JSON-RPC id.
- `send_notification/2` stamping is correct across payload shapes: a schema
  struct, an atom-keyed raw map, and a string-keyed raw map all end up with
  `subscriptionId` in `params._meta` in the matching key form.
- `send_notification/2` does **not** overwrite a pre-existing `subscriptionId`
  (or clobber other `_meta` keys), and is a no-op stamp when `request_id` is nil.
- `send_notification/2` stamps **only** subscription-stream notifications: with
  `request_id` set, a non-subscription payload (`notifications/progress` /
  `notifications/message` — struct, atom-map, or string-map) passes through
  **unstamped**, while a watched method (e.g.
  `notifications/subscriptions/acknowledged`) is still stamped.
- `Channel.request_id` is set for a normal request (e.g. `tools/call`) and nil
  for a notification-built channel.
