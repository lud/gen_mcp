# Release packaging + migration docs

Ship the fork. The packaging shape depends on the downstream client answer (feature-level
open question) but **does not block** building specs 002-005.

## Packaging decision (client-dependent)

Does any downstream client platform send different-version requests to the
**same** server endpoint? (Support ticket asked 2026-05-29.)

- **No** (likely -- each server speaks one negotiated version): keep a single
  `gen_mcp` package. `main` becomes the stateless `0.11` (or `1.0`); cut a
  **`0.10.x` maintenance branch** from the last stateful commit for backports
  to stragglers (`~> 0.10`). This is the TS-SDK model.
- **Yes** (one running server must field both protocols): Mix/Hex resolve one
  version per package name, so coexistence needs a **separate package**. Freeze
  the old spec as a legacy package; let `gen_mcp` evolve. Only adopt if
  confirmed -- avoid the duplication cost otherwise.

Record the outcome here once the client answer lands.

## Maintenance branch (if single-package)

- Branch `0.10.x` from the pre-fork commit.
- Document the backport policy: security/critical only; no features.

## Docs

- **CHANGELOG**: prominent breaking-change entry -- stateless core, removed
  sessions, `GenMCP.MCP` -> `GenMCP.MCP.V2607` (facade kept), dropped
  `session_*` callbacks, `Mcp-Session-Id` gone.
- **README**: rewrite "Supported Features" / "Roadmap" -- remove session/cluster
  bullets; describe the stateless model.
- **New guide** `guides/00X.stateless-and-state-handles.md`: explain the
  stateless model and the **explicit state handle** pattern (SEP-2567) as the
  replacement for server-side session state -- a tool mints a handle (e.g.
  `basket_id`), the model passes it back as a normal arg. This is the answer for
  users who previously relied on sessions.
- Update getting-started + suite guides to drop `session_controller` and the
  `session_*` callbacks.

## Semver

`0.x` -> next minor (`0.11`) is fine for the breaking fork; or jump to `1.0` if
you want to signal stability on the new spec. Note the draft is still moving --
prefer `0.11` until `2026-07-28` finalizes, then consider `1.0`.

### Possible solution: `v1` (stateful) + `v2` (stateless)

One option on the table (single-package model): release the current stateful
`main` as **`1.0.0`** and the new stateless core as **`2.0.0`**. The major bump
is the honest signal for a hard fork that deletes the session spine; `v1` is the
stable stateful lane, `v2` the stateless lane. The maintenance branch becomes
`1.x` (`~> 1.0`) instead of `0.10.x`.

Hard constraints on this approach (per the user):

- **Do not cut `1.0.0` / `2.0.0` until the new implementation is done AND the
  schema is fixed** -- i.e. `2026-07-28` is finalized and we are no longer
  vendoring `schema/draft`. While the draft moves, jumping to a major would
  spend the "`0.x` breaking changes are free" budget too early.
- **No version churn once `1.0`/`2.0` are out.** Until then the current `0.x`
  line stays live: a `0.11.x` / `0.12.x` can still land for security fixes or
  dependency-incompatibility fixes, but not for the fork itself.

Only viable if the packaging decision resolves to **single-package** ("No" to
the client question above): `v1`/`v2` of the *same* package cannot coexist at
one endpoint, since Mix/Hex resolve one version per package name.

## Acceptance criteria

- Packaging decision recorded once the client answer lands.
- CHANGELOG + README + guides updated for the stateless model.
- (If single-package) `0.10.x` maintenance branch exists with a documented
  backport policy.
