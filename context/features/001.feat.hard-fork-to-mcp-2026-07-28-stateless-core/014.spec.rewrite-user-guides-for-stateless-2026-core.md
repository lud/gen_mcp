# Rewrite user guides for the stateless 2026-07-28 core

The narrative guides under `guides/` predate the hard fork and still describe the
stateful, session-based library. They must be rewritten against the current
stateless core, the same way the module docs are being rewritten under the
`write-gen-mcp-docs` skill. This spec covers both the **tooling** to write guides
consistently and the **content** rewrite of the guides themselves.

The guides are ExDoc extras, registered in `mix.exs` `doc_extras/0` (grouped
under "Introduction" via `groups_for_extras/0`) and published to HexDocs.

### Readmix markers

At version-bump time `readmix/1` runs `Readmix` over `README.md` and every
`guides/**/*.md` (the `docs` recipe in the `justfile`), expanding the
`<!-- rdmx ... -->` markers. Two kinds matter for the rewrite:

- **`:eval`** -- Readmix evaluates code (e.g.
  `NimbleOptions.docs(GenMCP.Suite.init_opts_schema())`) and writes the result
  **between** the comment tags. That body is *generated*: do not hand-edit it, fix
  the source it derives from and let `readmix` regenerate it.
- **`:section name:... format:true`** -- a named code block that is *not*
  generated. You write and edit the code freely; `format:true` means `readmix`
  only reformats it. Because of that, **section code must be valid, compilable
  Elixir** (it gets formatted, and is the code users copy).

Any rewrite must keep these markers valid and balanced.

## Current guides (all stale)

- `guides/001.getting-started.md` (94 lines) -- install + quick start.
- `guides/002.using-mcp-suite.md` (308 lines) -- the Suite component model.
- `guides/009.system-configuration.md` (21 lines) -- logging/telemetry only.

Known staleness to fix (non-exhaustive; verify against current code, never trust
the old prose):

- **Session framing.** 002's intro sells the Suite as a way to "keep track of all
  incoming HTTP connections for a single session"; the stateless core has no
  sessions. Reframe around per-request handling and `GenMCP.Mux.Channel`.
- **Version marker.** 001 pins `{:gen_mcp, "~> 0.10"}` via the `:app_dep` rdmx
  marker. Confirm the marker still resolves to the right version after the fork's
  release decision (spec 006); the prose around it must not contradict it.
- **Capability list.** 002 says "Sampling and Elicitation are yet to come" -- the
  multi-round-trip / `Input*` work (spec 007) and subscriptions (spec 008) have
  since landed. State what exists now; do not catalog what is missing.
- **Behaviour shapes / helpers.** Re-verify every example against the current
  callbacks (`call/3`, `continue/3`, `ResourceRepo`, `PromptRepo`, `Extension`,
  subscriptions) and prefer the `GenMCP.MCP.V2607` helper constructors.
- **Discovery.** `server/discover` replaces `initialize`; reflect that wherever a
  guide describes how a client learns the server's capabilities.

## Part 1 -- Tooling: a guide-writing skill

Create a **new standalone skill** `write-gen-mcp-guides` (sibling of
`write-gen-mcp-docs`, living in `.claude/skills/write-gen-mcp-guides/`) with its
own `SKILL.md` and `guidelines.md`. It is NOT a generalization of the module-docs
skill: the procedures differ (a guide is a narrative document, not a worklist of
modules with `@moduledoc`/`@doc`/doctests).

**Keep (adapt from `write-gen-mcp-docs/guidelines.md`):** the reusable doc-craft
material that is independent of the @moduledoc/worklist machinery --

- House style: plain dash-free prose, no em/en-dashes, short sentences, ~90-col
  wrap, the "ride/rides on" ban, pattern-matching over access syntax in examples.
- "Document what exists, not what doesn't": no negative documentation, no rejected
  alternatives, positive/concrete directives naming real tools (`Phoenix.PubSub`).
- "Write examples the way a user would, or leave them out": `MyApp`/`:my_app`
  namespacing, no fabricated protocol plumbing, examples don't masquerade as
  library features.
- Accuracy from source; the golden rule that existing docs are stale and rewritten
  from the current code + `context/` specs.
- Protocol context (2026-07-28 stateless core) and the `GenMCP.MCP.V2607` helper
  rule.
- Cross-referencing/autolink conventions (backticks, `c:`/`t:` prefixes, fully
  qualified names) since guides autolink into the module docs.

**Drop / replace (module-doc-specific, not applicable to guides):** the worklist
procedure (`tools/docs-to-write.txt`, one-module-per-run, the "move to Documented"
step), the stale-doc strip sub-agent, `@moduledoc`/`@doc`/`@typedoc` mechanics and
the `@doc false` handling, the doctest-enabling test-module rules, the
per-callback / behaviour-module layout rules, the h3-inside-a-doc heading rule.

**Add (guide-specific):** how guides are registered (`mix.exs` `doc_extras/0`);
the `<!-- rdmx ... -->` marker conventions above (generated `:eval` bodies are not
hand-edited; `:section` code is hand-written but must compile so `format:true`
formatting succeeds); guide file naming (`NNN.slug.md`); cross-linking between
guides and into module docs; and a procedure suited to a narrative rewrite rather
than a module worklist. Decide and document whether guides get a worklist of their
own (the three files above, plus any new guide) so a run targets one guide at a
time.

## Part 2 -- Content: rewrite the guides

Rewrite each guide against the current code, following the new skill:

1. **001 getting-started** -- install, define a tool, mount the
   `GenMCP.Transport.StreamableHTTP` plug, hit the endpoint. Stateless framing.
2. **002 using-mcp-suite** -- the Suite component model (tools, resource repos,
   prompt repos, extensions, subscriptions), channel/assigns, per-request state.
   Drop the session narrative; update the capability list.
3. **009 system-configuration** -- logging via `GenMCP.attach_default_logger/1`
   and `GenMCP.TelemetryLogger`; expand with any other stateless-core
   configuration worth documenting.

### New guide: stateless model + state handles

explaining the stateless model and the explicit state-handle pattern (SEP-2567) as
the replacement for server-side session state. **That guide is owned here, not in

number.

## Relationship to spec 006

006 keeps the **release/packaging** decision (single-package vs separate package,
maintenance branch, semver) and the CHANGELOG/README updates that depend on it.
The **guide content** (rewriting 001/002/009 and the new stateless guide) moves
here. 006 is updated to point at this spec for the guide work.

## Acceptance criteria

- `write-gen-mcp-guides` skill exists with `SKILL.md` + `guidelines.md`, reusing
  the style material and dropping the module-worklist/doctest specifics.
- 001, 002, and 009 rewritten for the stateless core; no session framing; examples
  verified against current code and using `V2607` helpers.
- New stateless / state-handles guide written and registered in `doc_extras/0`.
- All `<!-- rdmx ... -->` markers valid and balanced; `:section` code compiles,
  `:eval` bodies regenerate cleanly via the `docs` recipe; `doc_extras/0` reports
  no unreferenced guides and `mix docs` renders the guides.
- Spec 006 updated to defer guide content to this spec.
