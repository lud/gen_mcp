# Transport coverage of the supported RPC surface

Close the gap between the **handler-dispatch layer** and the **wire-decode
layer**. A method can be fully wired in `GenMCP.Suite` (a `handle_request/3`
clause) and unit-tested there by building the struct directly — yet never be
reachable over HTTP, because the only thing that turns a wire message into that
struct is `GenMCP.Validator`, and the method may be missing from its `validable`
list (`lib/gen_mcp/validator.ex`). Nothing fails today because the Suite tests
bypass the transport entirely.

This surfaced with `subscriptions/listen`: `GenMCP.Suite.handle_request/3`
accepts `%MCP.SubscriptionsListenRequest{}` and is unit-tested
(`test/gen_mcp/suite/subscriptions_test.exs`), but the method is **not** in
`validator.ex`, so a real client POST is rejected with `{:unknown_method}`
before it reaches the Suite. The handler half is built; the transport half is
not connected. Treat that as the canary: where else does the supported handler
surface not line up with the wire surface?

## Goal

Every request and client-notification we claim to support must have at least one
**transport-level smoke test** in `test/gen_mcp/streamable_http_test.exs` — i.e.
driven as an HTTP POST through `GenMCP.Transport.StreamableHTTP`, exercising the
real `Validator.validate_request/1` → dispatch path. This guards against a
method being silently unreachable over the wire even though its handler is wired
and unit-tested.

These do **not** need to be thorough — each request's behaviour is unit tested at
the handler/Suite layer. One happy-path round-trip per method (assert it is
accepted, dispatched, and returns a well-formed JSON-RPC response / 202) is the
minimum.

## Supported surface (source of truth: the `validable` list in `validator.ex`)

Requests:

- `resources/list` (`ListResourcesRequest`) — **missing**
- `resources/templates/list` (`ListResourceTemplatesRequest`) — **missing**
- `resources/read` (`ReadResourceRequest`) — **missing**
- `prompts/list` (`ListPromptsRequest`) — **missing**
- `prompts/get` (`GetPromptRequest`) — **missing**
- `tools/list` (`ListToolsRequest`) — covered
- `server/discover` (`DiscoverRequest`) — covered
- `tools/call` (`CallToolRequest`) — covered

Notifications:

- `notifications/cancelled` (`CancelledNotification`) — covered

So the smoke tests to add now are the five missing requests above.

## `subscriptions/listen` — separate, larger gap (likely its own task)

`subscriptions/listen` is the motivating case but is **not just a missing test**.
It is missing from `validator.ex` *and* has no streaming dispatch path in the
transport (it opens a long-lived stream rather than returning a single result),
so wiring it end-to-end is a production-code change beyond adding a smoke test.
The result struct now exists (`SubscriptionsListenResult` /
`SubscriptionsListenResultMeta`, generated 2026-06-26). Decide with the user
whether this becomes a child task here or its own spec; the transport smoke test
for it follows once the wiring lands.

## Acceptance criteria

- A transport smoke test exists for each currently-supported request and
  client-notification in `validator.ex` (the five missing requests added).
- Each new test posts a real HTTP body and asserts the response round-trips
  (no `{:unknown_method}`, well-formed JSON-RPC result or 202).
- The `subscriptions/listen` reachability gap is captured (a note, plus a
  failing-or-skipped marker) so it is not forgotten.
- Tests stay minimal — depth lives in the per-layer unit tests.
