# Origin header validation (DNS-rebinding protection)

Deferred from spec 003 (2026-05-31). Needs a design decision; capturing the
background so it's actionable later.

## What the spec requires

From `draft/basic/transports` → Security & Endpoint:

- Servers **MUST** validate the `Origin` header on all incoming connections.
- If `Origin` is present and invalid → respond `403 Forbidden` (body **MAY** be
  a JSON-RPC error with no `id`).
- When local, servers **SHOULD** bind only to `127.0.0.1`, not `0.0.0.0`.
- Servers **SHOULD** implement authentication.

## The DNS-rebinding attack (the "learn this" part)

A local MCP server listening on `localhost` feels safe — only programs on the
machine can reach it. DNS rebinding defeats that:

1. The user visits `evil.com` in a browser. The page's JS wants to talk to the
   local MCP server at `http://localhost:port`.
2. The browser's same-origin policy normally blocks `evil.com` JS from reading
   responses from a *different* origin. But the attacker controls `evil.com`'s
   DNS: it first resolves to the attacker's server, then (after the page
   loads) **re-binds** the same hostname `evil.com` to `127.0.0.1`.
3. Now the page makes requests to `http://evil.com:port` — which the browser
   sends to `127.0.0.1:port` (your MCP server), and because the *hostname* is
   unchanged the browser treats it as same-origin and lets the JS read the
   responses.
4. The attacker's web page is now driving your local MCP server (calling tools,
   reading resources) from a remote site.

**The defense:** the server checks the `Origin` header. Legitimate local
clients either send no `Origin` (non-browser) or a known/allowed one; a
rebound browser request carries `Origin: http://evil.com`, which is not in the
allowlist → reject with `403`. The hostname the *browser* used cannot be
forged in `Origin`, so this catches the attack.

## Work (when picked up)

- Decide the allowlist policy: configurable allowed origins, plus the default
  (reject any present-and-unknown `Origin`; allow absent `Origin`?).
- Add it as a transport concern (likely an early check in the plug), so each
  `GenMCP` implementation does not re-implement it.
- Test: request with a disallowed `Origin` → `403`; request with no `Origin`
  (or an allowed one) → normal handling.
- Consider documenting the `127.0.0.1` bind recommendation for local servers.
