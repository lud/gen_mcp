# Stateless transport + remove the session machinery

The core of the fork. Make `StreamableHTTP` dispatch statelessly and **delete**
the session spine. This is the big simplification.

## Streaming model ‚Äî what SSE is for now (and what it is NOT)

Verified against `docs/specification/draft/basic/transports.mdx`
(<https://modelcontextprotocol.io/specification/draft/basic/transports>). The
2026-07-28 revision **removed the GET stream endpoint and protocol-level
sessions**, but **kept** Streamable HTTP's SSE. SSE is now a *per-request
transport concern*, not a session:

- For each JSON-RPC **request** POST the server chooses, **per request**, between
  `Content-Type: application/json` (a single direct response) or
  `Content-Type: text/event-stream` (an SSE stream carrying request-scoped
  notifications, then the final response). The client MUST support both.
- On an SSE stream the server **MAY** send `notifications/progress` /
  `notifications/message` related to the originating request before the result;
  the final response **SHOULD** terminate the stream. The server **MUST NOT**
  send JSON-RPC *requests* on any stream.
- So "always reply directly" (`application/json`) is a legal optimization **only**
  when a call emits no progress/log notifications. SSE the *mechanism* stays; the
  *session process* goes. In Elixir terms the SSE stream is just the Plug/Bandit
  request process streaming chunks for the life of one request ‚Äî no registered
  GenServer, no registry.

## Out of scope (moved to sibling specs)

  ‚Üí **spec 008** (draft).
- **MRTR / `InputRequiredResult` / `requestState`** ‚Äî server‚Üíclient

## Process model ‚Äî handle every request in a per-request GenServer

**Decided 2026-06-05.** Every JSON-RPC request is dispatched to a dedicated,
short-lived worker process (a GenServer-shaped process spawned per request). The
Plug/Bandit request process does **not** run the server callback inline; it
becomes a pure SSE relay: it `spawn_monitor`s the worker, then loops receiving
`{:"$gen_mcp", ...}` messages and writing them to the socket. The worker runs the
full `GenMCP` lifecycle (`init` ‚Üí `handle_request` ‚Üí optional `handle_message`
loop) and drives all output through `Channel`
(`send_progress`/`send_result`/`close`).

Why a process every time, not "inline for sync, spawn only when async":

- **Real-time progress requires producer ‚âÝ relay.** A tool emitting progress is a
  blocking function; while it runs, a *different* process must pump the socket.
  Running inline means `send_progress` just queues in the caller's own mailbox and
  never flushes until the call returns ‚Äî and if the call then returns a direct
  `{:reply, _}`, the notifications are silently lost. Always-spawn makes
  producer ‚âÝ relay unconditional, so `send_progress` just works anywhere.
- **One uniform path, no footguns.** `self()` is stable for the whole request, so
  monitors / registry / subscriptions can be set up anywhere (no split-`self()`
  between `init`/`handle_request` and a streamed continuation). It's the GenServer
  mental model Elixir devs already know.
- **Cost is negligible for this domain.** MCP traffic is LLM-driven: low QPS,
  latency dominated by the model and the tool's own I/O. A ¬µs-scale spawn never
  shows up in p50. The heavy state (Suite catalog) is built *inside* the worker by
  `init`, so it never crosses a process boundary; only the small request/opts are
  copied in.
- **Cleaner crashes.** The relay monitors the worker and converts an abnormal
  `:DOWN` into a proper JSON-RPC error instead of a generic Bandit 500.

Content-type is chosen **per request by the relay, from the first message**: first
message `:result`/`:error` ‚Üí `application/json`; first message a `:notification` ‚Üí
commit to `text/event-stream`, then stream the result at the end. The worker never
needs to know which mode it's in. This matches the spec's "server chooses per
request" wording.

Sync tools stay trivial to write ‚Äî still just `{:result, _}`; the worker/relay
machinery is invisible library plumbing the handler author never touches. If a
profile ever shows the spawn matters, the optimization is an **opt-in** inline
fast path for handlers that declare themselves pure (emit no notifications) ‚Äî
explicit, so it can never silently drop progress. Not building that now (YAGNI
until measured).

### Two handler paths + unified return tags (decided 2026-06-08)

Every callback return either **terminates** the request or **continues** it as a
stream. State rides **only** on the continue return ‚Äî a terminal return ends the
worker, so its state would be discarded and is not carried. The dividing line for
*which* path is: *does the handler need to receive arbitrary Erlang messages?*

- **Blocking (terminal)** ‚Äî compute (optionally emitting `send_progress` /
  `send_log`), then return `{:result, result}` or `{:error, reason}`. The common
  case. The worker owns its process, so it may block as long as it likes while
  the relay keeps pumping the socket. No `receive` block, no footgun.
- **Streaming (continue)** ‚Äî return `{:stream, state}` to hand control back to the
  worker's receive loop, carrying `state`. Subsequent Erlang messages are
  delivered to `handle_message/3` (the rename of `handle_info`), which shares the
  *same* return vocabulary. This is the path for handlers that subscribe to
  producers and emit output driven by external events ‚Äî the general-purpose
  long-lived listener (spec 008) is its canonical consumer.

`handle_request` and `handle_message` return the same set ‚Äî
`{:result, result}` | `{:error, reason}` | `{:stream, state}` (+ `{:stop, reason}`
on `handle_message` for a listener's no-result exit). One mental model: *terminate
or keep streaming.* See spec 004 for the full table.

**The channel is input-only.** It is immutable per-request framework context, so
it is **passed in** to every callback (for `send_progress`/`send_log`) but
**never returned** ‚Äî that is why the stream tag is `{:stream, state}`, not
`{:stream, channel, state}`. Handler-owned data lives in `state`; framework
context lives in the channel. (This is what lets the assigns system go ‚Äî see
*Channel becomes inert* below.)

**Why a callback loop instead of a hand-rolled `receive`.** A blocking `receive`
inside `handle_request` is a process that has left its OTP loop: it will not see
the relay's `:DOWN` (client disconnect) nor cooperate with system messages.
Returning `{:stream, state}` keeps the worker in the wrapper loop, which owns all
of that centrally and dispatches only user messages to `handle_message/3`. The
rule for handler authors: **anything that must receive arbitrary messages returns
`{:stream, ‚Ä¶}` ‚Äî never hand-roll `receive`.**

**Wrapper responsibilities** in the streaming path:

- **Monitor `channel.client`** (the relay). Its `:DOWN` means the client
  disconnected ‚Üí terminate the worker. This is the stateless-core cancellation
  story: there is no registry to find an in-flight request by id, but the relay's
  death is observable and sufficient. (A `notifications/cancelled` POST lands on a
  *different* worker with nothing to signal, so socket-close is the real signal.)
- Cooperate with system messages.
- Route every other message to `handle_message/3`.

**Eager SSE commit.** A streaming handler may sit idle before its first event
(e.g. a listener awaiting a subscription), so there is no "first message" to infer
content-type from. `{:stream, state}` therefore tells the relay to commit to
`text/event-stream` immediately. Blocking handlers still infer SSE lazily from the
first `send_progress`/`send_log` notification; the explicit tag is the "I'm
streaming, commit now even if silent" signal.

## Rework `lib/gen_mcp/transport/streamable_http.ex` (Impl)

- **Remove** `Mcp-Session-Id` entirely: `@session_id_header`,
  `lookup_session_id/1`, `reject_session_id/1`, `put_resp_session_id/2`,
  `terminate_session/2`, the `delete "/"` route's session deletion. Per the RC
  backward-compat rules, HTTP **GET and DELETE to the MCP endpoint ‚Üí `405 Method
  Not Allowed`**; a stray `Mcp-Session-Id` header is ignored (never minted or
  echoed); a `Last-Event-ID` header is ignored (streams are not resumable).
- **Version**: read the authoritative `MCP-Protocol-Version` HTTP header
  (SEP-1442); reconcile with `_meta` `io.modelcontextprotocol/protocolVersion`
  against the pinned draft. Single supported version -> reject mismatches with a
  clear error (no negotiation table).
- **Routing headers**: ~~parse `Mcp-Method` / `Mcp-Name` (validate-but-tolerate)~~
  enforced (400 + `-32001` on missing/mismatch) ‚Äî done via **task 009**. Origin
  validation (403, `allowed_origins` plug option) ‚Äî done via **task 010**.
- **Dispatch**: replace `Mux.start_session` / `ensure_started` /
  `Mux.request` with a direct call into the stateless runner (spec 004),
  building an ephemeral `Channel` (`Channel.from_request/2` already targets
  `self()` -- no registry). SSE streaming, keepalive, and the `stream_loop/1`
  machinery (lines 344-477) are **kept** -- they are transport concerns, not
  session concerns.
- Extract `_meta` `io.modelcontextprotocol/*` fields once per request into the
  channel (see *Channel becomes inert*): `progressToken`, `clientInfo`,
  `clientCapabilities`, `protocolVersion`, and `logLevel`. An unrecognized
  `io.modelcontextprotocol/logLevel` is rejected here with `-32602` (spec 011), so
  the channel only ever holds a valid level or `nil`.

## `GenMCP.Mux.Channel` becomes inert

The channel is now **immutable per-request framework context** ‚Äî an addressing +
read-only metadata handle (essentially a relay pid plus `_meta`), never threaded
as state and never returned from a callback.

**Re-scoped 2026-06-10:** the *breaking drops* move to **spec 004** because the
old Suite is their only consumer and it is rewritten there wholesale:

- ~~Drop `assigns`~~ ‚Üí **004**. 003 *adds* the write-once **`meta`** field
  (read-only request context from `_meta`); `assigns`/`assign/3` survive until
  the Suite rewrite deletes their last call sites (`extension.ex`, `tool.ex`
  docs).
- ~~Drop `status`/`set_streaming`/`as_closed` + the `send_*` `:closed` guards~~
  ‚Üí **004** (`suite.ex:198,495,550` are the consumers). Stream-state is already
  tracked by the wrapper and content-type by the relay on the new path.

**Done in 003:**

- **Keep** `from_request/2` reading `progressToken`; it now also reads
  `io.modelcontextprotocol/logLevel` into `log_level`, defaulting to **`nil`
  (logging disabled)** ‚Äî there is no library default level (spec 011).
- **Add `meta`**: `from_request/2` extracts `clientInfo`, `clientCapabilities`
  and `protocolVersion` from `_meta` `io.modelcontextprotocol/*` into a
  write-once `meta` field.
- **Keep** `send_progress` / `send_log` / `send_result` / `send_error` / `close`
  as the reply path. `send_log` is a **no-op when `log_level` is `nil`** (the
  client did not opt into logging ‚Äî MUST NOT emit; spec 011).

> Migration note: removing `Channel.assign/3` / `assigns` is breaking for anyone
> using them; fine for the 0.x hard fork, recorded in spec 006.

## Tests

- Rewrite the conn-case suite for stateless: `ping`, `tools/list`,
  `tools/call`, `resources/*`, `prompts/*`, `server/discover` round-trip with
  **no** session header and **no** `mcp-session-id` in responses.
- Update every handler stub to the unified return tags: terminal returns
  `{:result, _}` / `{:error, _}` (no state); streaming handlers return
  `{:stream, state}` and drive output from `handle_message/3` (renamed from
  `handle_info`), which returns `{:stream, state}` to continue or
  `{:result, _}` / `{:error, _}` / `{:stop, _}` to end. `handle_notification`
  returns `:ok`.

  progress/log notifications then the final result arrive on the SSE stream, and
  an error from a continuation terminates it.
